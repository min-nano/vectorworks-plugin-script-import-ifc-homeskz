# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## このリポジトリについて

**ホームズ君構造EX** から出力した木造軸組工法建築物の IFC ファイルをパースし、VectorWorks のオブジェクトに変換して配置することに特化した VectorWorks プラグインスクリプトです。

現在実装済みの機能は以下の通りです。

- グリッド線（通り芯）のインポート
- ストーリ・ストーリレベル・デザインレイヤの自動生成
- 横架材（土台・梁・桁）のインポート
- 柱（管柱・小屋束等）のインポート
- 床板（`IfcSlab`＝`床版`）のインポート（各階の `n-FL` レイヤに床ツール＝Floor オブジェクトで作図。床のある場所は IFC の床版外形から抽出し、厚みは 24mm 固定。**高さは IFC の床位置を尊重し、床の段差＝スキップフロアを表現する**＝床下端の絶対 Z を IFC 床版ソリッドの最下端そのままにし、標準の床高＝横架材天端レベルにバインドしてその高低差を `bound.offset` で表す。段差の無い床は横架材天端と一致＝offset 0、段差のある床や床高の異なる床は offset がずれる）
- 柱・横架材への構造クラス（`04構造-02木造-…`）の自動割り当て
- 基礎（立上り＝壁オブジェクト・底盤/地中梁＝スラブオブジェクト）のインポートと基礎ストーリの自動生成（同じ厚さ・同じ高さで連続する底盤は 1 枚のスラブに統合し、底盤の外周を立上りの外面＝壁心＋半壁厚まで広げる）
- 基礎の配筋（カスタム PIO `鉄筋`＝姉妹プロジェクト `vectorworks-plugin-rebar` への置換。立上り・地中梁＝梁モード、底盤＝スラブモード。配筋仕様は IFC の `Pset_Reinforcement` から取得し、無ければ既定値＝立上り・地中梁は上下 1-D13・せん断 D10@250、底盤は D13@150 シングル両方向）のインポート
- 交差する基礎立上り同士の壁結合（`JoinWalls` によるコーナー＝L・T 字＝T・十字＝X の結合。天端高さの異なる壁は低いほうを高いほうに `capped=True` で結合し、3 本以上が集まる交点・端点も適切に処理する）
- アンカーボルト（ハイブリッドシンボル `アンカーボルト_M12` / `アンカーボルト_M16` への置換）のインポート
- 床束（IFC に出力されないため大引の下に 910mm 間隔で決め打ち配置。ハイブリッドシンボル `床束` を基礎底盤上端の高さで専用レイヤ `F-床束` に配置）
- シートレイヤ（基礎伏図・各階の柱梁伏図・母屋伏図）の自動生成とビューポートの配置
- 基礎伏図のグラフィック凡例（VW 標準の「グラフィック凡例」PIO。基礎伏図ビューポートに表示されるシンボル＝既定ではアンカーボルトの凡例。M12＝`土台用アンカーボルトM12`・M16＝`ホールダウン用アンカーボルトM16` のラベルをコード内の固定マッピングで並べる。ソース定義＝シンボルをソースにし基礎伏図ビューポートでフィルタする設定はプラグインスタイル `基礎伏図凡例` に焼き込み、`SetPluginStyle` で関連付ける）
- 横架材への断面寸法データタグ（`断面寸法` スタイル）の自動配置（床伏図・小屋伏図・母屋伏図のビューポート注釈）
- 火打（火打梁）のインポート（ハイブリッドシンボル `鋼製火打` への置換）
- 仕口（受ける材のある横架材端部にハイブリッドシンボル `仕口` を配置。横架材の端部が別の横架材＝受ける材に取り付く場合、または柱の側面に取り付く場合にだけ、その端部＝梁端の中央上端に置く。取り付かない自由端には置かない。横架材と同じレイヤに描画）
- 柱・小屋束の span（またぐレベル区間）レイヤ分割（柱・小屋束を `n-柱` ではなく **`{from}to{to}-柱` レイヤ**に配置。`from`＝柱が立つ床レベル・1 始まり・GL=0、`to`＝上端が届く床/母屋レベル＝管柱は次階の整数・小屋束/屋根束は屋根面で止まる半整数 +0.5・通し柱は複数階ぶん上。各伏図はその切断レベルを span が含むレイヤだけを表示するため、下屋の小屋束＝例 `2to2.5`（to=2.5）が上階の小屋伏図＝切断 3.25 に写り込まない。通し柱＝例 `1to3` は跨ぐ各階の床伏図に自然に出る）
- 断面記号（実在する span レイヤごとに `柱束伏図記号` PIO を断面スタイル＝`MarkStyle=断面` で 1 つ重ね、そのレイヤ自身の柱・小屋束を実断面に合わせた記号＝柱×・小屋束/ で作図。対象クラスは絞らず（全クラス）、作図クラスは極細実線 `01作図-01線-02実線-01極細線`）
- （母屋伏図に小屋束を平面記号＝伏図記号で示す小屋束記号、および下階柱記号は span 方式で柱の表示自体をレイヤ切断レベルで制御するため役割が変わり、母屋伏図向けの小屋束伏図記号は移行後に別途対応する予定。現状 column_mark は断面記号のみを生成する）
- 垂木（IFC に出力されないため屋根版＝IfcSlab の屋根面の勾配・外形から導出。軸組ツール `FramingMember`（`type='rafter'`）で描画し、母屋の上に独立させた専用レイヤ `n-垂木` に配置。断面は既定 45×45・配置間隔は 455mm＝IFC に垂木寸法が無いための決め打ち。屋根版を持つ階ごとの母屋伏図に母屋・野地と併せて表示）
- 野地板（屋根版＝IfcSlab の屋根面 1 枚ごとに屋根ツール＝`BeginRoof` で単勾配の屋根オブジェクトを作図。垂木の上に独立させた専用レイヤ `n-野地板` に配置。厚みは 12mm 固定＝IFC に野地板固有の厚み情報が無いための決め打ち）
- 母屋伏図と柱梁伏図（床伏図・小屋伏図）の分離。複雑な下屋根では母屋等の小屋組と梁組が重なって見にくくなるため、**屋根版（屋根面）を持つ階ごとに専用の母屋伏図（シートレイヤ・ビューポート）を作り**、小屋組（母屋・垂木・野地板）を梁組と分けて表示する（母屋伏図は母屋の上からの見下げ図なので小屋束の断面は出さず、小屋束の位置は平面の伏図記号で示す＝移行後に別途対応）。最上階（屋根）の主屋根だけでなく、中間階に架かる下屋根（下屋）にも 1 枚ずつ作る（母屋・棟木・垂木・野地板は梁と別レイヤ `n-母屋`／`n-垂木`／`n-野地板` に分離済み）。図面名は屋根が架かる階番号を付けた `n階母屋伏図`（例 2 階建てで主屋根＝`2階母屋伏図`、下屋根＝`1階母屋伏図`）。母屋伏図には**屋根を貫いて立ち上がる主屋の柱（管柱・通し柱）を切断レベル（その階の床レベル+0.75。1 階母屋伏図＝2.75・2 階母屋伏図＝3.75）を span が含む柱レイヤで表示する**（母屋を支える小屋束の断面は切断より低いため出さない）。柱梁伏図には小屋組を重ねず、図面名も `n階床伏図`・`n階小屋伏図`（母屋の表記なし）にする

## アーキテクチャ: 2 フェーズ分離

処理は **IFC 解析フェーズ** と **VectorWorks 描画フェーズ** に完全分離されている。両フェーズは JSON 直列化可能な**命令セット（ドキュメント）**だけで接続され、`vs` との密結合を避けることで検証や VectorWorks バージョンアップ対応を容易にしている。

1. **IFC 解析フェーズ（`ifc` サブパッケージ）** — `vs` に一切依存しない。ifcopenshell で IFC を解析し、描くべきオブジェクトを命令セット（dict）として組み立てる。通常の Python 環境で単体実行・検証できる。
2. **描画フェーズ（`vw` サブパッケージ）** — `vs` だけに依存し、IFC・ifcopenshell の知識を持たない。命令セットを検証（`validate_document`）してから vs API で描画する。

命令セットのスキーマ（version・stories/grids/members/rafters/roofs/columns/walls/slabs/floors/rebars/anchor_bolts/floor_posts/fire_braces/joints/sheets/tags/legends 各命令の形式）は `document.py` の docstring に定義されている。スキーマを変更するときは `DOCUMENT_VERSION` の互換性に注意し、`validate_document()` とテストも併せて更新すること。`run()` は両フェーズの間で `json.dumps`/`json.loads` を通すため、命令セットに直列化不能なオブジェクト（ifcopenshell エンティティや vs ハンドル等）を入れてはならない。

## コーディング規約: 型注釈

すべての関数・メソッド（テストコード・モック用クロージャ含む）に引数と戻り値の型注釈を付ける。型検査は mypy で行い、CI で `mypy` を実行する（設定は `pyproject.toml` の `[tool.mypy]`、`disallow_untyped_defs` 有効）。

- 各モジュール先頭に `from __future__ import annotations` を置く。Python 3.9 互換を保ちつつ `list[str]` / `X | None` 構文を使うため。
- 命令セットの型は `document.py` の `TypedDict`（`Document` / `StoryCommand` / `GridCommand` / `MemberCommand` / `RafterCommand` / `RoofCommand` / `ColumnCommand` / `WallCommand` / `WallJoinCommand` / `SlabCommand` / `FloorCommand` / `RebarCommand` / `AnchorBoltCommand` / `FloorPostCommand` / `FireBraceCommand` / `JointCommand` / `ColumnMarkCommand` / `SheetCommand` / `ViewportCommand` / `TagCommand` / `LegendCommand` / `LegendItemCommand` / `LevelCommand` / `StoryBoundCommand`）を使う。`GridCommand` / `MemberCommand` / `RafterCommand` / `RoofCommand` / `ColumnCommand` / `WallCommand` / `SlabCommand` / `FloorCommand` / `ColumnMarkCommand` / `RebarCommand` は `class` キー（作図クラス名）が予約語のため functional 構文で定義している（`WallJoinCommand` / `AnchorBoltCommand` / `FloorPostCommand` / `FireBraceCommand` / `JointCommand` / `SheetCommand` / `ViewportCommand` は `class` キーを持たないため通常の `class` 構文で定義する）。スキーマ変更時は `TypedDict` 定義・docstring・`validate_document()` を同時に更新すること。
- `ifc` サブパッケージでは ifcopenshell の型（`ifcopenshell.file` / `ifcopenshell.entity_instance`）を注釈にのみ使う場合 `if TYPE_CHECKING:` ブロックで import する。
- `vs` モジュールは型スタブが存在しないため `ignore_missing_imports` で許容し、vs ハンドルは `Any` で扱う。VectorWorks 公式 `vs.py` スタブ（`tests/vs.py`）は型検査対象から除外している。
- 検証前の命令セット（JSON 由来の信頼できない入力）を受ける関数（`validate_document()` / `execute_document()`）の引数は `Any` とし、検証済みの値だけを `Document` 型として扱う。

## スクリプトの実行方法

このスクリプトは単独の Python プログラムとして動作しません。**VectorWorks 内でプラグインスクリプトとして実行する必要があります**。`vs` モジュールは VectorWorks 独自の Python スクリプト API であり、pip でインストールすることはできません。

テストは VectorWorks の公式 `vs.py` スタブをモック対象として `pytest` で実行します（`.github/workflows/test.yml` 参照）。

## 外部ライブラリの利用方法

VectorWorks の組み込み Python は pip パッケージを標準では参照しませんが、Python Externals フォルダは VectorWorks が自動的に `sys.path` に追加します。このため以下の手順だけで外部ライブラリを利用できます。

1. `pip install --target <Python Externals フォルダ> .` でパッケージ（および依存ライブラリ）を Python Externals フォルダにインストールする。
2. VectorWorks から呼び出される `main.py`（ラッパー）が `vectorworks_plugin_import_ifc_homeskz.run()` を呼び出す。

Python Externals フォルダのパスは OS・VectorWorks のバージョンによって異なります（詳細は `README.md` 参照）。新しい外部ライブラリへの依存を追加するときは `pyproject.toml` の `[project] dependencies` に記載してください。

## IFC 解析の方針

IFC の解析には **`ifcopenshell`** を利用します。生の STEP テキストの正規表現マッチではなく、エンティティと属性を辿る形でデータを抽出します。

ただし**読み込み時のサニタイズ**は例外的にテキスト処理を行います（`ifc/loader.py` の `open_ifc`）。ホームズ君 EX の IFC2X3 出力には IFC4 でのみ定義される `IfcFootingType`（STEP の `IFCFOOTINGTYPE`）が混入しており、Python 3.9 で唯一解決される `ifcopenshell==0.8.4.post1` はこの不正エンティティにつまずいて周辺の正常な `IfcFooting`・`IfcSlab` まで取りこぼす（基礎が 1 件しか読めない）。`open_ifc` は**サニタイズが必要な古い ifcopenshell（バージョン < 0.8.5）の場合のみ**、解析前にスキーマ非適合のエンティティ行を除去してから `ifcopenshell.file.from_string` で開く。不具合が解消された 0.8.5 以降（Python 3.10+）では余計なテキスト処理を避け `ifcopenshell.open` に委ねる（バージョン判定は `_needs_sanitize`）。除去対象は基礎の型エンティティのみで、本スクリプトは `IfcFooting` の型を参照しないため抽出結果に影響しない。`run()` とテストのフィクスチャ読込はこの `open_ifc` を経由する。

## スクリプトの処理フロー

`vectorworks_plugin_import_ifc_homeskz.run()` は以下の順で処理を行います。

1. **ファイル選択** — `vs.GetFileN()` でファイルダイアログを開き `.ifc` を選択。
2. **IFC オープン** — `open_ifc(filepath)`（`ifc/loader.py`）でファイルを読み込む。解析前にスキーマ非適合のエンティティを除去する（基礎の取りこぼし防止、下記「IFC 解析の方針」参照）。
3. **解析（フェーズ1）** — `ifc.build_document(ifc_file)` で JSON 命令セットを組み立てる。
4. **JSON 経由の受け渡し** — `json.dumps` → `json.loads` を通し直列化可能性を保証。
5. **描画（フェーズ2）** — `vw.execute_document(document)` が検証後、ストーリ → 通り芯 → 構造材 → 垂木（軸組ツール）→ 野地板（屋根ツール）→ 柱 → 立上り（壁）→ 壁結合（`JoinWalls`）→ 底盤/地中梁（スラブ）→ 床板（床ツール）→ 配筋（鉄筋 PIO）→ アンカーボルト（シンボル）→ 床束（シンボル）→ 火打（シンボル）→ 仕口（シンボル）→ 断面記号（柱束伏図記号 PIO・柱の span レイヤごと）の順で描画し、デザインレイヤのスタック順を `reorder_story_layers` で整えたあと、最後にシートレイヤ（ビューポート）・横架材の断面寸法データタグ（ビューポート注釈）・基礎伏図のグラフィック凡例（シートレイヤ上の PIO）を描画して実行数を返す。**並べ替えで床・野地板レイヤを最背面へ回した結果は、インポート直後のビューポート描画には反映されない**（レイヤパレット・ビューポートのプロパティ上は新しい順＝床が最下段なのに、描画だけ古い順のまま床が柱梁を覆い隠す）。VW 上の検証で、並べ替え（`HMoveForward`）は既存ビューポートを out-of-date にせず、`UpdateVP`・`ReDrawAll`・`Project 2D` トグルによる強制再描画もスクリプト内では効かないことを確認した（**手動「ビューポートを更新」またはファイルの開き直しでのみ反映される**）。スクリプト内での自動反映は VW の制約上できないため、余計な更新処理は行わず、この描画反映はユーザーの手動操作に委ねる（VW の制約として許容）。シート・タグ・凡例はデザインレイヤと横架材オブジェクトを参照するため、レイヤ生成・並べ替え・構造材描画の完了後に描画する。構造材描画（`execute_members`）は配置した横架材ハンドルをインデックス付きで記録し、シート描画（`execute_sheets`）がタグをこのハンドルに関連付ける。構造材描画（`execute_members`）は配置した横架材ハンドルをインデックス付きで記録し、シート描画（`execute_sheets`）がタグをこのハンドルに関連付ける。壁結合（`execute_wall_joins`）は立上り描画（`execute_walls`）が記録した壁ハンドルを結合命令のインデックスで引くため、立上りをすべて配置した直後に実行する。断面記号（`execute_column_marks`）は配置後のリセットで対象の span 柱レイヤの柱・小屋束を検索するため、柱描画（`execute_columns`）の後に置く。
6. **完了ダイアログ** — `vs.AlrtDialog` で結果を表示。

### 通り芯（ifc/grid.py → vw/grid.py）

- 解析: `IfcGridAxis` を走査し `AxisCurve`（`IfcPolyline`）の端点を取得。ソート済みタプルキーで同一ジオメトリの重複線を除去し、バウンディングボックスの中心 `(center_x, center_y)` でセンタリングした座標・レイヤ名・クラス名を grid 命令に格納。
- 描画: 各命令を `vs.CreateCustomObjectPath('GridAxis', ...)` で生成。失敗時は通常の線にフォールバック。レイヤ（`共通`）は存在しなければ作成。

### 横架材（ifc/member.py → vw/member.py）

- 解析: `IfcBeam` / `IfcMember` を走査し、配置・断面寸法・材種から member 命令（座標は通り芯と同じ中心オフセットで補正、構造材 ID は `{幅}×{背} - {材種}`）を組み立てる。一般階は `横架材天端` レイヤ、最上階は `軒高` レイヤを指定。軸（`Axis`）が鉛直な材は横架材でないためスキップする。
  - **母屋・棟木の分離**: 構造クラスが `CLASS_MOYA` / `CLASS_MUNAGI` の材（＝名前で判別した母屋・棟木、および最上階で名前で判別できず軒高超えで母屋と推定した材）は、小屋梁・軒桁と重なって見にくいため配置先の横架材天端（最上階は `軒高`）レイヤではなく専用の `n-母屋` レイヤに配置し、高さ基準も `母屋` レベルにバインドする。`母屋` レベルは横架材天端（最上階は `軒高`）と同じ絶対 Z（`offset=column_offset`、最上階は 0）なので `offset` の算出（天端 Z − レベル絶対 Z）は横架材と変わらない。ルーティングはクラス判定（`resolve_member_class`）の後に行い、母屋の実描画高さは天端バインドが担う（レベルのオフセットには依存しない）。**最上階の主屋根だけでなく、中間階に架かる下屋根（下屋）の母屋・棟木も同様に分離する**（`story.py` が該当階に `母屋` レベルを作る条件＝名前判定 `story_has_moya` と一致させる。中間階の母屋クラスは名前判定のみで付くため、レイヤ生成と振り分けが齟齬なく一致する）。この分離により屋根版を持つ階ごとの母屋伏図（最上階の主屋根・中間階の下屋根、下記シート節）に母屋・棟木を分けて出力できる。
  - 基準点補正: ホームズ君 IFC の配置点（ローカル配置 Z）は**断面中心**だが、VW 構造材ツールの断面基準点は**左右中央・上端（天端中央）**。このため断面中心線を軸に直交し軸を含む鉛直面内で上向きの単位ベクトル方向に `背/2` 持ち上げた**天端中央線**を命令に格納する（水平梁では単純に Z + 背/2）。`elevation`／`end_elevation` は始端／終端の天端 Z（絶対値、`ストーリ高さ + ローカル Z + 補正`）。レイヤの基準高さに固定しないため、基準高さにない梁（段差のある梁等）も正しい高さに描画される。Z を取得できない梁のみレイヤ基準高さ（`ストーリ高さ + resolve_beam_top_offset`、最上階はストーリ高さ。いずれも既に天端なので背/2 補正なし）にフォールバックする。
  - 高さ基準のバインド（`start_bound`／`end_bound`）: 柱と同じく構造材ツールの始端／終端の高さ基準を配置先レイヤのストーリレベル（一般階は `横架材天端`、最上階は `軒高`）に `SetObjectStoryBound` でバインドする。横架材は自階のレベルに乗るため `story_offset=0`。`offset` は**バインド先レベルの絶対 Z から天端 Z（`elevation`／`end_elevation`）までの距離**（平らな梁は ≈0、段差梁は一定値、傾斜梁は始端／終端で異なる値）。これをしないと構造材ツールの高さ基準が `レイヤの高さ`・オフセット 0 のまま `Move3D` で動かした実ジオメトリと矛盾し、オブジェクト編集時に高さが 0 にリセットされてしまう。**梁の高さ（傾斜を含む）はこのバインドだけで表し、パスには Z 成分を持たせない**（下記「描画」参照。持たせると傾斜が二重加算される）。
  - 傾斜梁（登り梁・隅木・谷木等）: `Axis` 属性の Z 成分から始端・終端の高さ差を求め、`elevation ≠ end_elevation` の傾斜した命令にする。平面座標は軸の XY 成分 × 全長（= 平面投影長）で求める。傾斜（`end_elevation − elevation`）は描画時に始端／終端の高さバインドの `offset` 差として表れ、パスは水平のまま。
  - 食い込み調整（`resolve_member_interferences`）: 命令組み立て後、横架材同士が食い込んでいる箇所（甲乙梁の T 字や出隅の L 字の取り合い等）の端部を相手梁の面まで詰めて干渉を解消する。ある梁の端点が別の梁の矩形に入り（端部も含む）、配置レイヤが一致し Z 範囲（`[天端 - 背, 天端]`）が重なる場合が対象。**勝ち負けの判定**: 自分の端点が相手に食い込む量 `sAB` と、相手の端点が自分に食い込む量 `sBA` を比べ、`sAB > sBA + _SYMMETRY_TOL`（自分の方が深く食い込む＝相手が通し材で勝ち）のときだけ自分を詰める。相手梁の形状は変えず負け側だけを短くする。相互の食い込み量が同等な**対称の角**（同寸の出隅・火打等で端点が一致するケース）は勝ち負けが付かないため触らない。平行（同一直線上の継ぎ手）・Z が離れた段差梁・面ちょうどで止まる端部（既に勝ち負けが入った角）・傾斜梁（高さが一定でなく水平面内の矩形モデルが成り立たないため、詰める側にも相手側にもしない）は対象外。判定は入力時点のジオメトリに対して行い命令の並び順に依存しない。
- 描画: 構造材ツール `vs.CreateCustomObjectPath('StructuralMember', ...)` で配置（**パスは平面投影**でローカル原点から作成し `Move3D` で始端天端の絶対位置へ移動）。配置後 `SetObjectStoryBound(obj, 0, 2, …)`（始端）・`SetObjectStoryBound(obj, 1, 2, …)`（終端、`boundType=2`=Story）で高さ基準を `start_bound`／`end_bound` のストーリレベルにバインドする（柱と同じ規約）。**傾斜梁も含め、始端／終端の高さ（傾斜）はこのバインドだけで表し、パスには Z 成分を持たせない**（`elevation`/`end_elevation` の差分をパスの Z にも持たせると、構造材ツールの高さバインドがパス由来の部材長に加算されて傾斜が二重に適用され、終端が実際の 2 倍の高さに描画される。柱の二重加算（#54）と同種の問題で、水平梁は差が 0 のため顕在化しなかった）。配置後 `vs.SetPluginStyle(obj, '木質構造材_横架材')` でプラグインスタイルを関連付ける。その後に `MemberID` 等の個別フィールドを設定するためスタイルの既定パラメータは実測値で上書きされる。**`SetPluginStyle` はスタイルの関連付け（パラメータ）までで、スタイルが決める描画属性（コンポーネントのクラス／マテリアル＝テクスチャ等）はオブジェクトへプッシュされない**（OIP での手動適用はリンク＋スタイル更新の両方を行うが、`SetPluginStyle` は前者のみ。#56 の不具合）。描画属性は `execute_members` が**全配置後に `vs.UpdateStyledObjects('木質構造材_横架材')` を 1 回**呼び、当該スタイルの全オブジェクトをスタイルから更新して反映する（by-instance の個別フィールド＝寸法・`MemberID` 等は保持したまま by-style の描画属性のみ更新される。per-object の `UpdatePIOFromStyle` は Python バインディングが引数なしで対象オブジェクトが曖昧なため使わない）。配置先レイヤが存在しない命令はスキップし、プラグインが使えない場合は通常線にフォールバック。

### 垂木（ifc/rafter.py → vw/rafter.py）

ホームズ君 EX の IFC には**垂木が一切出力されない**（オブジェクト・型・プロパティのいずれにも垂木の位置や仕様が現れない。全フィクスチャで確認済み）。そのため垂木は IFC から抽出できず、要件どおり**屋根版（`IfcSlab` の屋根面。`Name` が `屋根版` 始まり）の勾配・外形から導出**する。垂木を VectorWorks の**軸組ツール `FramingMember`（部材種別 `type='rafter'`）**で描き、母屋の上に独立させた専用レイヤ `n-垂木` に配置する。断面は既定 **45×45**・配置間隔は **455mm**（IFC に垂木寸法が無いための決め打ち。`ifc/rafter.py` 冒頭の `DEFAULT_RAFTER_WIDTH`/`DEFAULT_RAFTER_HEIGHT`/`RAFTER_INTERVAL` に集約）。

- 解析（`build_rafter_commands`）: FL ストーリを走査し、各ストーリに含まれる `屋根版` の `IfcSlab` から垂木を導出する。屋根版は勾配した平面外形を鉛直に押し出したソリッド（押し出し＝屋根の厚み）で、基礎・火打と同じ `ifc/footing.py` の `_world_solid` でワールド情報に変換する。平面外形の頂点列（`origin + lx*u + ly*v`）と面の単位法線（配置の局所 Z 軸 `lz` を上向きに揃えたもの）が屋根面。座標は通り芯・横架材と同じグリッド中心オフセットで補正する。ストーリのローカル配置は XY=0・Z=Elevation なので、平面座標はそのまま（中心オフセットのみ）・天端 Z はストーリ Elevation を足して絶対値にする（横架材の `storey_elevation + oz` と同じ規約）。差し込み（`embedment`）に使う桁幅は横架材命令から相互参照するため、`build_document` が一度だけ組み立てた `members` を渡す（`build_rafter_commands(ifc_file, members)`。未指定なら内部で `build_member_commands`）。
  - **勾配方向・掃引（`_rafters_for_plane`）**: 屋根面の法線の水平成分 `d`＝最急勾配方向（母屋・棟木・軒桁に直交）。垂木はこの方向に沿って軒側（低い端）から棟側（高い端＝`end`）へ架かる。`d` に直交する掃引方向 `e`（軒・棟に平行）へ、面の広がりの**両端の垂木を屋根面の端から垂木幅の半分だけ内側**へ寄せ（端に軸を合わせると垂木幅の半分がはみ出すため）内部は 455mm 以下で掃引線を並べ、各掃引線を屋根面の外形で走査線法によりクリップした区間を 1 本の垂木にする（非凸の面＝ドーマ等も正しく分割）。区間の平面投影長が極小（`_MIN_RAFTER_LENGTH`＝隅木際の極小片等）のものは配置しない。天端 Z は屋根面（平面）の式から求める（法線の符号反転に対して不変）。
  - **始点（支持点）・軒の出・差し込み・ラベル**: `start` は**屋根面（垂木下端）が横架材天端（最上階は軒高）の Z レベル `beam_top_z` と交わる点＝軒（横架材高さ）の中心線との交点（支持点）**にする（`beam_top_z`＝一般階は `Elevation + resolve_beam_top_offset`、最上階は軒高＝`Elevation`）。屋根面の下端（軒先）と棟を結ぶ線上で `z=beam_top_z` となる位置を求め、そこを支持点、`elevation` をその `beam_top_z` にする。**差し込み `embedment` は支持点の真下にある軒桁（横架材命令）の桁幅の半分**（`_girder_width_at`＝支持点に芯線が最も近く垂木に直交する横架材の幅。見つからなければ既定桁幅 `DEFAULT_GIRDER_WIDTH` の半分）で、支持点→壁外面の水平距離を表す。**軒の出 `overhang`（壁外面から軒先の距離）は支持点→軒先の水平距離から `embedment`（支持点→壁外面）を引いた残り**（軒先が既に `beam_top_z` 以上で支持点が取れない面は `start`＝軒先のまま `overhang=0`）。VW の垂木は軒先を **支持点＋差し込み（`bearinginset`）＋軒の出（`overhang`）** の位置に置く（実オブジェクトのエクスポートで確認）ため、`embedment + overhang` が支持点→軒先に一致するよう軒の出から差し込みぶんを引き、軒先が支持点→軒先の位置に揃うようにする（引かないと VW の既定差し込み 88.9mm＝3.5inch のぶん軒先が外へずれる）。`label` は垂木の仕様ラベル（`{幅}×{背}@{間隔}`＝`45×45@455`）。
  - **配置先レイヤ**: 屋根版を含むストーリの母屋レイヤの直上に独立させた `n-垂木`。最上階（屋根）の主屋根だけでなく、中間階に架かる下屋根（下屋）の屋根版も同様に、その階の `n-垂木` に配置する。**下屋根は母屋を持たないこともある**（単純な片流れ下屋等）ため、垂木レイヤの有無・振り分けは母屋（`story_has_moya`）ではなく**屋根版の有無**（`story_has_roof`／`collect_story_roof_flags`）で判定する（`story.py` が該当階に `垂木` レベルを作る条件と一致させる）。命令の高さ情報（`elevation`/`end_elevation`）は屋根面由来の絶対 Z で、レベルのオフセットには依存しない。
- 描画（`vw/rafter.py` の `execute_rafters` / `draw_rafter`）: 命令の軒側 `start`（支持点）・棟側 `end` の平面座標と両端の天端 Z から**水平投影長（`LineLength`）・平面方位角・勾配（`pitch`）**を求める。軸組ツールを `vs.CreateCustomObjectN('FramingMember', (0,0), 0, showPref=False)` で原点に生成し、`ResetOrientation3D` → `Rotate3D(0, 0, 方位角)`（軒→棟）→ `Move3D(支持点 XY, 支持点天端 Z)` で支持点の絶対位置へ配置する。勾配（垂木の傾き）は本体の `pitch` パラメータが担い、支持点（下端基準 `verticalReference='bottom'`）から棟側へ立ち上がる。`vs.SetRField` で `type`/`width`/`height`/`LineLength`/`pitch`/`verticalReference` に加え、**2D 表示＝「幅」（`2DDisplay`＝`_2D_DISPLAY_WIDTH`）・軒の出（`overhang`）・差し込み（`embedment`→ VW フィールド `bearinginset`。実オブジェクトのエクスポートで確認した登録名で、既定 88.9mm を上書きする）・仕様ラベル（`label`）・構造用途＝垂木（`StructuralUse`）・材質＝木（`Material`）**を設定し `vs.ResetObject` で反映、`vs.SetClass` で構造クラス（`04構造-02木造-05小屋組-05垂木`＝`CLASS_TARUKI`）を付ける。**フィールド名・値（軒の出・差し込み・ラベル・構造用途・材質の各フィールド名、2D 表示・構造用途・材質の値）は VectorWorks の FramingMember 登録に一致させる必要があり、軸組ツールの高さ／向き／`pitch`・各パラメータの最終挙動は VectorWorks 上で検証する方針（他要素と同じ。`vw/rafter.py` 冒頭の名前付き定数に集約）**。配置先レイヤが存在しない命令はスキップし、プラグインが使えない場合は平面投影の直線にフォールバックする。実行順は構造材（横架材）の後・柱の前（`execute_document`）。
- **伏図への表示**: `n-垂木` レイヤは母屋・野地板と併せて母屋伏図に表示する。屋根版を持つ階ごとの母屋伏図（最上階の主屋根＝`R-母屋`・`R-垂木`・`R-野地板`、中間階の下屋根＝`n-母屋`・`n-垂木`・`n-野地板`）に出力する（`ifc/sheet.py`。母屋は母屋がある場合のみ、垂木・野地板は屋根版がある場合に表示する。下屋根に母屋が無くても垂木・野地板は表示する）。**構造クラス `04構造-02木造-05小屋組-05垂木`（小屋組の葉クラス）・PIO 内部名 `FramingMember`・部材種別 `rafter` は VectorWorks 側の登録に一致させる必要がある**（`vw/rafter.py` 冒頭の定数に集約）。

### 野地板（ifc/roof.py → vw/roof.py）

野地板（屋根の下地合板）を要件どおり **VectorWorks の屋根ツール（`vs.BeginRoof`）** で描画する。野地板は屋根面そのものに沿う 1 枚の面材なので、垂木と同じ**屋根版（`IfcSlab` の屋根面。`Name` が `屋根版` 始まり）1 面ごと**に、その勾配・外形から単勾配の屋根オブジェクトを作図する。要件により**厚みは 12mm 固定**（IFC に野地板固有の厚み情報が無いための決め打ち。`ifc/roof.py` 冒頭の `NOJIITA_THICKNESS` に集約）。屋根版 1 面 = 野地板 1 枚（垂木のように 455mm 間隔で分割はしない）。

- 解析（`build_roof_commands`）: FL ストーリを走査し、各ストーリに含まれる `屋根版` の `IfcSlab` 1 面ごとに屋根（roof）命令を組み立てる。屋根面のジオメトリ（平面外形頂点列 + 単位法線）は垂木（`ifc/rafter.py`）と同じ `_roof_plane` を再利用する（いずれも屋根版から屋根面を取り出す）。座標は通り芯・垂木と同じグリッド中心オフセットで補正する。ストーリのローカル配置 Z（=Elevation）を足して絶対 Z にする。**軒の目標 Z（`elevation`）は屋根版の平面（＝垂木下端。VW 上の実測で確認）から垂木せい（`DEFAULT_RAFTER_HEIGHT`=45）＋野地板厚（`NOJIITA_THICKNESS`=12）を鉛直換算（÷cosθ＝単位法線の鉛直成分 `nz`。垂木・野地板は勾配があるため屋根面に直交する寸法を鉛直へ勾配補正する）して持ち上げた値**にする（野地板は垂木の上に載る＝野地板下端＝垂木上端）。
  - **軒（屋根軸）・upslope・勾配（`_roof_command_for_plane`）**: 屋根面の法線の水平成分 `d`＝最急勾配方向。`+d` は天端 Z が下がる軒側。軒に平行な方向 `e`（`d` に直交）へ、footprint を `d` 方向へ射影した**最下点（最も軒側）**を通り `e` 方向に footprint の広がりぶん伸ばした線分を屋根軸（`axis_start`／`axis_end`＝`BeginRoof` の p1/p2）にする（footprint 全体が軸の棟側＝upslope 側に来るようにする）。`upslope` は軸から棟側へ向かう定義点。勾配は屋根面の単位法線の水平成分 `dh`・鉛直成分 `nz` から `rise=dh`・`run=nz`（slope=rise/run=tanθ）。天端の絶対 Z（軒の高さ）を `elevation` に持たせる。ほぼ水平な面（法線が鉛直）・広がりが極小の面は命令を作らない（垂木と同じ扱い）。
  - **配置先レイヤ**: 屋根版を含むストーリの垂木レイヤの直上に独立させた `n-野地板`。最上階（屋根）の主屋根だけでなく、中間階に架かる下屋根（下屋）の屋根版も同様に、その階の `n-野地板` に配置する（垂木と同じく屋根版の有無 `story_has_roof`／`collect_story_roof_flags` で判定＝`story.py` が該当階に `野地板` レベルを作る条件と一致させる）。命令の高さ情報（`elevation`）は屋根面由来の絶対 Z で、レベルのオフセットには依存しない。
- 描画（`vw/roof.py` の `execute_roofs` / `draw_roof`）: 配置先レイヤ（`n-野地板`）をアクティブにし、屋根ツールで屋根オブジェクトを作る。**呼び出し列は VW 上で屋根を作成したドキュメントの VectorScript エクスポートに一致させる**（実エクスポートで確認した正規手順。#113）。ただし**屋根オブジェクトの座標系は作図レイヤ基準（レイヤ相対）**であることが VW 上の検証で確認された（絶対 Z で高さを与えるとレイヤ高さぶん余計に持ち上がり本来の約 2 倍の高さに描画される）ため、高さはレイヤ相対（`軒の絶対 Z − レイヤの Z`）で扱う: ① `vs.GetZVals()` で作図レイヤの Z/ΔZ を退避し `vs.SetZVals(レイヤ Z, 厚み)` で厚み（ΔZ＝野地板厚 12mm）を予約する（**厚みはこの ΔZ が担う。`SetRoofAttributes` は使わない**。Z はレイヤの値を維持＝ストーリレベルにバインドされたレイヤでは Z の変更が効かないため依存しない）→ ② `vs.BeginRoof(p1, p2, upslope, rise, run, miter, vertPart)` で軒（軸）・upslope・勾配を与える（勾配はエクスポートの規約どおり **run=25.4＝1 インチあたりの rise** に正規化。`vertPart`＝厚み×sinθ もエクスポートと同じ計算）→ ③ **`EndGroup` の前、`BeginRoof` 直後に** `vs.Move3D(0, 0, 軒のレイヤ相対 Z)` を呼ぶ（エクスポートと同順。実測では適用されないが害はない）→ ④ 2D 図形（屋根の水平投影外形の閉じたポリゴン）をテンプレートとして描き `vs.EndGroup()` で確定 → ⑤ `vs.SetZVals` を退避値へ復元（以降のフェーズへ影響を残さない）→ ⑥ **高さの自己補正**: 確定した屋根の実際の軸 Z（`GetRoofFaceCoords` の `Zaxis`、レイヤ相対）を実測し、目標のレイヤ相対 Z との差分だけ `vs.Move3D` で移動する（屋根はレイヤ平面＝バインド先ストーリレベルの高さ＝軒高・地廻りに作られるため、補正しないと母屋下がりの屋根面が地廻り高さに張り付く）。確定後の屋根への後付け操作は `vs.SetClass`（`04構造-02木造-06耐力面材-03屋根`＝`CLASS_ROOF_SHEATHING`）のみ。**確定後の屋根に `SetRoofAttributes`（`CreateRoof` の屋根コンテナ用）・`Move3DObj`・`ResetObject` を呼んではならない**（エクスポートに現れない後付け操作で、未定義動作により VW 本体がクラッシュする。#113）。**屋根が作れたかは `LNewObj` の前後比較＋`GetTypeN` のタイプ判別で確認する**: テンプレートとして描いた外形ポリゴン自体が `LNewObj` で返るため NIL 判定では `BeginRoof` の失敗を検出できず、失敗時に残ったポリゴンや直前の別オブジェクト（垂木 PIO 等）へ屋根専用の設定を呼ぶと未定義動作でクラッシュする（#113）。屋根と確認できないオブジェクトには屋根専用の設定を行わず、外形ポリゴンのフォールバック（テンプレートのポリゴンが残っていればそれを使う）にとどめる。配置先レイヤが存在しない命令はスキップする。実行順は垂木の後・柱の前（`execute_document`）。屋根の実描画（高さ・勾配・厚みの最終挙動）は他の要素と同じく VectorWorks 上で最終確認する方針（`vw/roof.py` 冒頭の名前付き定数に集約）。

### 柱（ifc/column.py → vw/column.py）

**柱は梁と同じ構造材ツール（`StructuralMember`）で鉛直材として描く。** 拡張パッケージの柱・間柱ツール（`柱・間柱` / AAPillarS）はスクリプト操作に対して不安定なため、標準の構造材ツールに置き換えている。これにより伏図記号・柱頭/柱脚金物専用フィールドといった柱・間柱ツール固有機能は使えなくなるため、後述の通り扱いを変えている。構造材ツールの**構造用途（`StructuralUse`）は種別で分ける**：管柱・通し柱は**柱**（`'4'`）、小屋束は**小屋束**（`'5'`）。小屋束を柱用途で描くと VW が柱の高さモデルを適用し、上端の高さオフセットと部材長が矛盾して上端高さが正しく描画されないため、小屋束用途にして実ジオメトリどおりの高さで描く（`StructuralUse` の値は VW の構造用途プルダウンの並び順に対応：`<自動>, 梁='1', 桁, 根太, 柱='4', 小屋束='5', …`）。**柱・小屋束の高さはパスのジオメトリ（下端の絶対 Z + 柱高さ）で確定させたうえで、上下端を横架材天端（最上階は `軒高`）のストーリレベルに `SetObjectStoryBound` でバインドする**（梁と同じ仕組み）。**柱（管柱・通し柱）は下端を当階の横架材天端、上端を上階（次階、`story_offset=1`）の横架材天端に、小屋束は下端・上端とも当階の横架材天端（`story_offset=0`）にバインドする**。`offset` は基本的に**バインド先レベルの絶対 Z から実ジオメトリ（パスで配置した下端／上端の絶対 Z）までの距離**。**ただし VW の構造材ツールは上下端 story bound の `offset` 差（top − bottom）をパス由来の部材長に加算する**（かつて `resolve_height_bounds` が同一レベルに柱高さぶんの `offset` を与えパス由来の部材長と加算して上端が二重になった #54 と同じ挙動）。柱（管柱・通し柱）は上端を上階にバインドするため `offset` 差が ≈0 になり二重加算は生じないが、**小屋束は上端も当階の横架材天端にバインドするため、上端 `offset` を天端相当（`bottom + height`）にすると柱高さが二重加算され上端が約 2 倍になる。そこで小屋束は上端 `offset` を下端と同値にして加算分を 0 にし、上端高さはパス（下端 + `height`）に委ねる**（#116 のフォローアップ）。バインドにより構造材ツールの高さ基準が `レイヤの高さ`・`offset 0` のまま実ジオメトリと矛盾して編集時に高さがリセットされるのを防ぐ。

- 解析: `IfcColumn` を走査し、配置・断面寸法（`IfcRectangleProfileDef` の `XDim`/`YDim`）・柱高さ（`IfcExtrudedAreaSolid.Depth`）から column 命令を組み立てる。XY 座標は通り芯と同じ中心オフセットで補正。**柱は span（またぐレベル区間）ごとの専用レイヤ `{from}to{to}-柱`（命令の `layer`）に配置する**。`from`＝柱が立つ床レベル（1 始まり＝ストーリ index+1・GL=0）、`to`＝上端が届く床/母屋レベル（`resolve_column_to_level`）。`to` は上端を base より上の各階の**横架材（床梁）下端**（`members` から算出。母屋は含めず横架材天端/軒高レイヤの梁だけ）と比べて到達した最上階（`reached`）を求める：直上階の梁下端に達しなければ屋根束として `from+0.5`（下屋の小屋束・棟束・主屋根束）、達すれば管柱として `from+1`、さらに上の階の梁下端まで達すれば通し柱としてその階まで（1・2 階通し柱＝`1to3`）。天端でなく**下端**を境界にするのは、通常の管柱が横架材天端でなく下端まで（梁を下から受ける高さ）しか来ないため（天端を境界にするとホームズ君のモデルで天端付近まで伸びた管柱を通し柱と誤判定する）。**ただし到達階の横架材天端（最上階は軒高）より上端が高い柱は、その階の床/軒に下から受けられる管柱・通し柱ではなく屋根面で止まる屋根束なので `reached+1+0.5` の半整数レベルにする**（`beam_tops`＝各階の横架材天端の絶対 Z を渡す。これがないと、軒高より高く突き出す小屋束が屋根軒高の梁下端に達しただけで管柱扱いになり、`1to2.5` に振り分けるべき柱が `1to2` になってしまう。管柱・通し柱の上端は受ける梁の下端＝天端より梁背ぶん下に止まるため天端を超えずこの分岐には来ない）。柱は梁と同じ構造材ツールで描き上下端をストーリレベルにバインドするため、span レイヤ自体の高さ（レベルオフセット）には依存しない。
- 高さ（上下端）: **パスのジオメトリで確定させ、ストーリレベルにバインドする**。`elevation`（柱下端の絶対 Z = `ストーリ高さ + ローカル配置 Z`）と `height`（柱高さ = 押し出し `Depth`）から、描画フェーズが鉛直パスをローカル `(0,0,0)〜(0,0,height)` で作り `Move3D` で下端の絶対位置へ配置する。上端は `elevation + height`。加えて命令は上下端の高さ基準（`bottom_bound`／`top_bound`＝`StoryBoundCommand`）を持ち、下端は当階の横架材天端（最上階は `軒高`、`story_offset=0`）、上端は柱（管柱・通し柱）なら上階（次階、`story_offset=1`）の横架材天端、小屋束なら当階の横架材天端（`story_offset=0`）にバインドする。`offset` は原則**バインド先レベルの絶対 Z から実際の下端／上端の絶対 Z までの距離**（各階の横架材天端の絶対 Z は `ストーリ高さ + resolve_beam_top_offset`、最上階は `軒高`＝ストーリ高さ）。**VW は上下端 `offset` 差をパス長に加算するため、小屋束だけは上端 `offset` を下端と同値にして二重加算を避ける**（上記参照。柱は上端が上階バインドで `offset` 差 ≈0 のため問題ない）。
- 構造用途（`structural_use`）: 割り当てクラスが小屋束（`CLASS_KOYAZUKA`）なら小屋束用途 `'5'`、それ以外（管柱・通し柱）は柱用途 `'4'` を命令に格納する。描画フェーズが `StructuralUse` に設定する。小屋束を柱用途にすると上端高さが崩れるため用途を分ける。
- 構造材 ID（`member_id`）: `{幅}×{成} - {種別}` を基本とし、柱頭・柱脚金物の仕様（空でないもの）を ` / ` 区切りで連結する（`make_column_member_id`）。構造材ツールには金物専用フィールドが無いため、金物仕様は `MemberID` に含めて保持する。種別は `IfcColumn.ObjectType`（`None` または `STANDCOLUMN`）を `resolve_column_type` で変換した名前（`None`→`管柱`、`STANDCOLUMN`→`小屋束`、未知の値は `管柱`）。断面は現状すべて矩形。
- 柱頭・柱脚金物: ホームズ君 IFC では柱頭・柱脚金物が柱と同じストーリに含まれる `IfcMechanicalFastener`（柱頭/柱脚付近の立方体）として表現される。名前に `柱頭金物`／`柱脚金物` を含む金物を、柱と**同じ平面座標**（XY を丸めたキー）で柱に対応付け、金物の型 `IfcMechanicalFastenerType` の名前（例: `柱頭金物:(ろ)`・`柱頭金物:C12`）を**加工せずそのまま**仕様文字列として `top_hardware`／`bottom_hardware` に格納する（該当金物が無ければ空文字）。これらは構造化した記録として命令に個別保持しつつ、`member_id` にも連結して `MemberID` で保持する。ホームズ君側で金物定義をカスタマイズしている場合、型名が想定形式とは限らないため、コロン分割等の加工で文字列が失われる（空欄になる）のを避けて型名全体を登録する。型は IfcRelDefinesByType 経由で辿り、逆方向属性名がスキーマで異なる（IFC2X3=`IsDefinedBy`／IFC4=`IsTypedBy`）ため両方を走査する。
- 描画: 構造材ツール `vs.CreateCustomObjectPath('StructuralMember', path, profile)` で配置（梁の描画と同じ規約）。鉛直パスはローカル原点 `(0,0,0)` から `(0,0,height)` で作り、`Move3D` で柱下端の絶対位置（XY + `elevation`）へ移動する。続いて `SetObjectStoryBound(obj, 0, 2, …)`（下端）・`SetObjectStoryBound(obj, 1, 2, …)`（上端、`boundType=2`=Story）で高さ基準を `bottom_bound`／`top_bound` のストーリレベルにバインドする（梁と同じ規約）。**`offset` はパスで配置した実ジオメトリと一致するため二重加算にならず**（同一レベルへ柱高さぶんの `offset` を与えて加算し上端が二重になった #54 とは異なる）、編集時に高さがレイヤ基準へリセットされるのを防ぐ。断面は `width`×`depth` の矩形プロファイル。構造用途は命令の `structural_use`（柱=`'4'`／小屋束=`'5'`）を `StructuralUse` に設定する。`MemberID` に `member_id` を格納し、その他のレコードフィールド（`ProfileShape`/`MajorBreadth`/`MajorDepth`/`B`/`D`/`MemberType` 等）は梁の構造材と同じ値を設定する。配置後 `vs.SetPluginStyle(obj, '木質構造材_柱・束')` でプラグインスタイルを関連付け、その後に個別フィールドを設定するためスタイルの既定値は実測値で上書きされる。**`SetPluginStyle` はパラメータの関連付けまでで描画属性（コンポーネントのクラス／マテリアル＝テクスチャ等）はプッシュされない**ため、描画属性は `execute_columns` が**全配置後に `vs.UpdateStyledObjects('木質構造材_柱・束')` を 1 回**呼んで反映する（梁と同じ規約。#56）。配置先レイヤが存在しない命令はスキップし、プラグインが使えない場合は断面の矩形にフォールバック。

### ストーリ（ifc/story.py → vw/story.py）

ホームズ君 IFC の高さ表現ルールを利用してストーリを構築します。

- 名前が `FL` で終わる `IfcBuildingStorey` のみを対象とする（`設計GL` 等の参照高は除外）。`IfcBuildingStorey.Elevation` がそのまま VectorWorks の**ストーリ高さ**になる（例: `1FL=473.0`, `2FL=3273.0`, `RFL=5973.0`）。
- 最上階以外の階では、`IfcRelContainedInSpatialStructure` を辿って `IfcColumn`・`IfcSlab` のローカル配置 Z 座標（負値、例: `-48.0`, `-36.0`）を集め、その**最大値**（床に最も近接した 0 以下の値）を**横架材天端**の相対オフセットとして使用します（エンティティ列挙順に依存しない決定的な結果にしつつ、床に最も近い横架材天端を採用するため）。
- ストーリ名は `1階`, `2階`, ..., `屋根`（最上階は常に `屋根`）。
- ストーリ suffix（前/後 記号）は `"1"`, `"2"`, ..., `"R"`（最上階は `R`）。空文字 suffix は VW 2026 で 2 回目以降の `CreateStory` が失敗するため不可。
- デザインレイヤ名は `1-FL`, `1-横架材天端`, `1-柱`, ..., `R-軒高`, `R-母屋`, `R-垂木`, `R-野地板`, `R-小屋束`, `R-柱`（接頭辞はストーリ suffix と一致）。下屋根を持つ中間階は `n-母屋`・`n-垂木`・`n-野地板` も持つ。
- ストーリレベル: 一般階は `FL`(0) と `横架材天端`(負値)、最上階は `軒高`(0)。加えて各階に**柱・小屋束配置用の span レベル**（`{from}to{to}-柱` レイヤ。その階を base=from とする span レイヤを `(from, to)` 昇順で `column_layers_by_story` が渡す）を持つ（高さは `横架材天端`＝最上階は `軒高` に揃える。level type はレイヤ名で一意。柱の上下端はバインド先レベル名で決まり span レベルのオフセットには依存しない）。span 方式では従来の下階柱記号（`下階柱` レベル）・小屋束記号（`小屋束` レベル）は廃止した（柱の表示は伏図の切断レベルで制御する。母屋伏図向けの小屋束伏図記号は移行後に別途対応）。**母屋（棟木含む）配置用の `母屋` レベル**（`LEVEL_MOYA`、`n-母屋` レイヤ）は、最上階（屋根）は常に、中間階は下屋根（下屋）の小屋組（母屋・棟木）を名前判定（`story_has_moya`／`collect_story_moya_flags`）で含む場合に持ち、横架材天端（最上階は `軒高`）レベルの**直前**に挿入してそのレイヤの直上に積む（高さは横架材天端＝最上階は `軒高` に揃える＝`offset=column_offset`、最上階は 0。母屋の分離は横架材節参照）。**垂木配置用の `垂木` レベル**（`LEVEL_TARUKI`、`n-垂木` レイヤ）・**野地板配置用の `野地板` レベル**（`LEVEL_NOJIITA`、`n-野地板` レイヤ）は、最上階は常に、中間階は屋根版（屋根面）を含む場合（`story_has_roof`／`collect_story_roof_flags`）に持ち、`垂木` は `母屋` レベルの直上（母屋が無ければ横架材天端／軒高の直上）、`野地板` は `垂木` レベルの直上に積む（高さは母屋＝横架材天端／軒高に揃える。実描画の Z は屋根版由来の絶対値で垂木・野地板の要素が持つ。**下屋根は母屋を持たなくても屋根版＝垂木・野地板があるため、母屋とは別に屋根版の有無で判定する**。垂木節・野地板節参照）。挿入は横架材天端（最上階は軒高）レベルの直前（`tail`）位置に `母屋` → `垂木` → `野地板` の順で行い、後から挿入したものが上段に来るため、スタックは上→下で 野地板 → 垂木 → 母屋 → 横架材天端／軒高 になる。`levels` の並び順は**希望するデザインレイヤのスタック順（上→下）**を表し、span 柱レイヤを `n-FL`（最上階は `R-軒高`）レイヤの直上に積むため span レベルを先頭（`(from, to)` 昇順）に置く。
  - レイヤのスタック順はレベルの高さに縛られず作成後に並べ替えできる（VW のナビゲーションで `#` 列をドラッグするのと同じ）。`AddLevelFromTemplate` はレイヤをレベルの高さ順に挿入するため、放置すると `柱`(横架材天端の高さ)が `FL` の下に入り、さらにストーリ間の並びも入り乱れる。これを描画フェーズ（`vw/story.py` の `reorder_story_layers`）が `HMoveForward(layer, False)` の 1 段ずつの移動で**全レイヤを 1 本の希望スタック順**（`desired_layer_order`）に揃える。希望順（ナビゲーション上→下）は **`共通`（通り芯）を最上段**に、続いて**最上階→最下階**の順に各ストーリの**構造レイヤ**を並べ、**最後に全ストーリの床（`FL`）・野地板レイヤをまとめてスタック最下段（背面）へ回す**（`desired_layer_order` の `_BACKGROUND_LEVEL_TYPES`）。床・野地板を背面へ集めるのは、シートレイヤの伏図ビューポートで床オブジェクト・野地板が柱・梁・柱記号を覆い隠さないようにするため（**ビューポート単位のレイヤ重ね順オーバーライドは VW の API に無い**＝ビューポートは常にドキュメントのデザインレイヤ重ね順で描画される。実オブジェクトの VectorScript エクスポートでも、ビューポートのレイヤはドキュメント重ね順でしか列挙されず重ね順を設定する関数は現れないことを確認済み）。かつ伏図には**下階（別ストーリ）の柱レイヤも重ねて表示する**ため、床・野地板を各ストーリ内で下げるだけでは足りず（上階の床は下階の柱レイヤより前面に来てしまう）、ドキュメント全体の最下段へ移す必要がある。各ストーリの構造レイヤの順は `levels` 順から床・野地板を除いたもの（一般階が `柱` → [`下階柱`] → [`小屋束`] → [`垂木`] → [`母屋`] → `横架材天端`、最上階が `柱` → `下階柱` → `小屋束` → `垂木` → `母屋` → `軒高`。最下階は `下階柱` を持たず、`小屋束`・`垂木` は最上階と屋根版を持つ中間階、`母屋` は最上階と下屋根の母屋を持つ中間階）。最下段の床・野地板も最上階→最下階の順（上ほど前面）に並べる（例: `共通, R-柱, R-下階柱, R-小屋束, R-垂木, R-母屋, R-軒高, 2-柱, 2-下階柱, 2-小屋束, 2-垂木, 2-母屋, 2-横架材天端, 1-柱, 1-横架材天端, R-野地板, 2-FL, 2-野地板, 1-FL`）。命令は Elevation 昇順（最下階→最上階）なので逆順に辿る。隣接ペアを末尾（下）から先頭（上）へ処理し、確定済みの下のペアを崩さずに上のペアを揃える。`共通` レイヤは通り芯描画フェーズで生成されるため、`reorder_story_layers` は `execute_stories` 内ではなく `execute_document` が**全描画（ストーリ→通り芯→構造材→柱）の後**に呼ぶ。まだ生成されていないレイヤ（通り芯が無い場合の `共通` 等）は `GetLayerByName` が NIL を返すため自動的にスキップされる。`HMoveForward`／`HMoveBackward` の第 2 引数 `toFront`／`toBack` を `True` にすると**レイヤが削除される**（公式ドキュメントの注意書き）ため、必ず `False` で 1 段ずつ移動する。レイヤ走査（`FLayer`→`NextLayer`）は下→上の順で、`NextLayer(anchor)` が対象レイヤになった時点（対象が anchor の直上）で停止する。

上記はすべて解析フェーズで決定され story 命令（`name`/`suffix`/`elevation`/`levels`）に格納される。描画フェーズはそれを実行するだけで判断ロジックを持たない。

VW 2026 でレイヤをストーリレベルに正しくバインドするには、`AddStoryLevelN` + `AssociateLayerWithStory` ではなく `CreateLevelTemplateN` + `AddLevelFromTemplate` の組み合わせを使う必要がある（前者ではバインドが UI 上 `<なし>` になる）。`create_story_level_via_template()` の処理順:

1. `vs.CreateLevelTemplateN(desired_layer_name, 1.0, level_type, elevation, 2400.0)` — テンプレート登録（戻り値の index を保持）
2. `vs.AddLevelFromTemplate(story_handle, index)` — ストーリにレベル追加 & レイヤ自動生成 & バインド
3. `vs.GetLayerForStory(story_handle, level_type)` — 生成されたレイヤのハンドル取得
4. `vs.SetName(layer_h, desired_layer_name)` — `AddLevelFromTemplate` が末尾に付ける suffix（例: `1-FL-1`）を取り除いて意図した名前にリネーム

ストーリ作成順序: `CreateStory` → `SetStoryElevationN` を 1 ストーリ毎に実行してから次のストーリへ。`SetStoryElevation` を後回しにすると全ストーリが既定高さ 0 で衝突して 2 階以降の `CreateStory` が失敗する。

基礎要素が存在する場合、`build_document` は `ifc/footing.py` の `build_foundation_story_command` が返す **基礎ストーリ** を stories の**先頭**（最下層）に追加する（`build_story_commands` が返す FL 階は変更しない）。基礎ストーリは `name=基礎`・`suffix=F`・`elevation=0`（GL は常に 0）で、レベルは `基礎天端`（立上り天端の絶対 Z、`F-アンカーボルト` レイヤ）・`GL`(0、`F-立上り` レイヤ)・`床束`（底盤天端の絶対 Z、`F-床束` レイヤ）・`底盤天端`（底盤天端の絶対 Z、`F-底盤` レイヤ）の 4 つ。`levels` の並びは希望スタック順（上→下）で、最上段にアンカーボルト（`基礎天端`）、続いて立上り（`GL`）・床束（`床束`）・底盤（`底盤天端`）を積むため `基礎天端` → `GL` → `床束` → `底盤天端` の順にする。Elevation=0 で最下層になり、`reorder_story_layers` の希望スタック順（最上階→最下階）でも最下段に積まれるため、`1-横架材天端` の下に `F-アンカーボルト`・`F-立上り`・`F-床束`・`F-底盤` が並ぶ。アンカーボルトの高さ基準は `基礎天端` レベル、床束（シンボル）の高さ基準は `床束` レベル（底盤上端に揃える）が担う。

### 基礎（ifc/footing.py → vw/footing.py）

ホームズ君 IFC の基礎要素（`IfcFooting` と基礎底盤の `IfcSlab`）を `Name` で 3 種に分類し、別オブジェクトに変換する。座標は通り芯・横架材と同じグリッド中心オフセットで補正する。配置先レイヤが存在しない命令はスキップする。

- **立上り（基礎梁、`Name` が `基礎梁` 始まり）→ 壁オブジェクト（wall 命令、`F-立上り` レイヤ）**。矩形断面の幅を壁厚、背を壁高とし、壁芯は配置原点から押し出し方向の線。下端は基礎（自階）の `GL`(`story_offset=0`)、上端は 1 階（上階）の `横架材天端`(`story_offset=1`)にバインドする。`offset` は実形状の下端/上端の絶対 Z とバインド先レベルの絶対 Z（1 階横架材天端 = `1FL Elevation + resolve_beam_top_offset(1FL)`）の差。描画は `vs.DoubLines`(壁厚)→`vs.Wall`(壁芯)→`vs.SetWallOverallHeights`(下端/上端とも `boundType=2`=Story、story 引数は 0=自階/1=上階)。**高さ基準は汎用の `SetObjectStoryBound` ではなく壁専用の `SetWallOverallHeights` を使う**: 壁は `SetObjectStoryBound` では高さ基準が確定せず、デザインレイヤの「壁の高さ（レイヤ設定）」（Default Wall Height）に従ってしまう（構造材・スラブでは `SetObjectStoryBound` が効くが壁は専用関数が必要）。`SetWallOverallHeights` で下端/上端を直接ストーリレベルにバインドすることでレイヤの壁高さ設定に依存せず実形状どおりの高さになる。`SetWallOverallHeights` の story 引数（0=自階・1=上階・2=下階）は命令の `story_offset`（0/1）とそのまま一致する。壁生成後 `vs.SetWallStyle(obj, '基礎 - 木造ベタ基礎150mm', 0.0, 0.0)`（「壁 基礎立ち上がり」フォルダ内のウォールスタイル、オフセット 0/0 で壁芯に揃える）を適用する。壁が作れない場合は壁芯の直線にフォールバック。
  - **立上りのマージ（`merge_wall_commands`）**: ホームズ君 IFC では立上りが通り芯の交点等で細かく分断されており、そのまま描くと壁オブジェクトが多数に分かれる。**同一直線上にあり同一断面形状（壁厚・上下端の高さ基準が一致）の立上り**を 1 本の壁に統合して、できるだけ分断のない形状で描く。`build_wall_commands` は wall 命令を組み立てた後にこの統合を通す。断面キー（`_wall_section_key`＝レイヤ・クラス・壁厚・下端/上端の `story_offset`/`level`/`offset` を許容値 `_WALL_MERGE_DIST_TOL`=1mm で丸めたもの）ごとにグループ化し、各グループ内で Union-Find（`_merge_wall_group`）により**同一直線上で区間が重なる/接触する立上り**（`_walls_connected_collinear`: 方向が平行・端点が相手の壁芯直線上・射影区間が重なる/接触する）の連結成分をまとめ、成分ごとに先頭の壁芯方向へ全端点を射影した最小〜最大区間の 1 本にする（高さ基準・壁厚・クラスは先頭のものを引き継ぐ）。**断面が違う（壁厚・高さの違う）立上り、同一直線上でも隙間がある立上り、平行だが別の線上（直交距離が壁厚分ある側並びの壁）にある立上りは統合しない**（隙間を橋渡しして実在しない壁を作らない）。統合はグループ化・成分処理とも入力順に対して決定的で命令の並び順に依存しない。地中梁（`thickness=None` のスラブ）はマージしない（底盤のマージは下記「基礎（底盤）」節参照）。
  - **自由端の半壁厚延長（`_extend_free_wall_ends`）**: ホームズ君 IFC では**他の立上りと交差しない端点**は柱芯までの長さで入力されているが、実際の基礎立上りはそこから**半壁厚だけ長い**（端面が柱芯より半壁厚外側にある）。`build_wall_commands` はマージの後にこの補正を通し、どの立上りとも交差しない端点だけを壁芯（自分の軸）方向へ **`壁厚/2`** 外側に延長して実形状に合わせる。交差する端点（コーナー・T 字等）は相手壁の外面までモデル化済みで既に半壁厚のオーバーハングを持つ（`_wall_intersection` の端点許容参照）ため触らない。交差の有無は `_wall_intersection` で判定し、交点が壁芯の端点にある（`a_at_end`／`b_at_end`）とき、交点に最も近い端点を「交差する端点」として延長対象から除外する。延長は各壁の自軸に沿って外側（始点は始点側・終点は終点側）へ行うため、交差しない側並び・平行の関係を新たに作らず、壁結合（`build_wall_join_commands`）の交点判定にも影響しない（自由端は元々どの交点にも関与しない）。判定は入力順に対して決定的。
  - **立上りの壁結合（`build_wall_join_commands` → `execute_wall_joins`）**: マージ後に**交差する立上り同士**を VW の壁結合（`vs.JoinWalls`）で結合し、コーナー・T 字・十字の取り合いをきれいに繋ぐ。2D での正確な表現のため、天端高さと交点に集まる本数を考慮して結合する。命令は結合する 2 壁の **walls 内インデックス** `a`／`b`・壁芯の交点 `point`（参照用）・各壁に渡すピック点 `pick_a`／`pick_b`・結合種別 `join_type`（`JoinWalls` の joinModifier：1=T・2=L・3=X）・結合部を閉じるか `capped`（`JoinWalls` の capped 引数）を持つ。
    - **交点のジャンクション化（`_wall_junctions`）**: 全ペアの壁芯交点（`_wall_intersection`）を求め、交点が `_JOIN_CLUSTER_TOL`（=1mm）以内で近いものを 1 つのジャンクション（同一交点に集まる立上りの集合）にまとめる（union-find）。3 本以上が 1 点に集まる場合、その 3 本の全ペアの交点は数学的に同一点になるため 1 つのジャンクションに束ねられる。**同一交点あたり 1 本のバックボーンへ枝を繋ぐスパニングツリー**にし、全ペアを冗長に結合しない（3 本なら結合は 2 件）。
    - **端点／内部の判定**: 交点が各壁芯の**端点か内部か**を `_wall_point_at_end` で判定する（端からの距離が**半壁厚＋`_JOIN_ENDPOINT_TOL`（=1mm）以内**なら端点）。**端点許容に半壁厚を含めるのは、ホームズ君 IFC の立上りが直交する相手壁の外面までモデル化され、コーナーで壁芯どうしの交点が壁の端から半壁厚（例 120mm 厚なら 60mm）離れるため**。固定 1mm 許容だと外周コーナー（L）が「両方とも内部で交わる」＝X 結合と誤判定され VW の `JoinWalls` がコーナーを繋がない（`#80` の残課題）。半壁厚を含めることで外面で終わる／手前で止まるコーナーを正しく端点交差（L／T）と判定し、区間内判定でも取りこぼさない。同一直線上（平行）の立上りは merge が 1 本に統合済みで結合対象にしない（`_wall_intersection` が平行を除外）。
    - **結合種別（`_emit_junction_joins`）**: ジャンクションに**内部で交わる通し壁がある**場合は、天端が最も高い通し壁をバックボーンにし、他の通し壁を**X 結合**、端点で突き当たる壁を**T 結合**（端点側の stem＝`a`、通し側 through＝`b`。`JoinWalls` は firstWall を secondWall まで延長する。through は stem に最も直交する通し壁を選ぶ）にする。**通し壁が無い端点コーナー**は、天端高さ降順で**はじめの 2 本を L 結合**、それ以降を**T 結合**（バックボーンへ突き当てる。要件3）にする。
    - **天端高さと `capped`（要件1・要件2）**: 結合する 2 壁の**天端高さ（`top_bound.offset`。基礎立上りは全て 1 階横架材天端にバインドするため offset だけで比較できる）が `_WALL_HEIGHT_TOL` を超えて異なる**ときは、**低いほうの壁を高いほうに結合**（低い壁を `a`）して**端部を閉じる（`capped=True`）**。同じ高さの立上り同士はコンクリートで一体のため閉じない（`capped=False`）。3 本以上が集まる交点では、天端が最も高い立上り同士を `capped=False` で先に繋ぎ、それより低い立上りを `capped=True` で繋ぐ（命令は `capped=False` を先に並べる）。判定は入力順に対して決定的。
    - **ピック点（`pick_a`／`pick_b`、`_kept_side_pick`）**: `JoinWalls` に渡すピック点は各壁の**「残す側」＝交点から遠い端点の方向へ寄せた壁芯上の点**にする。**壁芯の交点そのものは相手壁の壁芯上にも乗るため「どちら側を残すか」が曖昧になり、VW が L 結合でコーナーを詰めず立上りが相手壁の外面まで伸びたまま残る（外周の出隅・入隅が結合されない `#84` の不具合）**。交点から残す側へ寄せることで残す区間を明示する。寄せ量は交点に集まる立上りの**最大壁厚**（相手壁の footprint＝半壁厚を確実に越える）を基準にし、交点〜遠い端点の距離の `_PICK_OFFSET_FRAC`（=0.4）までにクランプして、**詰める端点（近い側）が最も近い端点のまま**保たれる（寄せすぎて遠い端点が最寄りになると VW が残す／詰める側を取り違える）ようにする。T 結合の through・X 結合の両壁は VW が詰めないため寄せは無害。解析フェーズ（`_emit_junction_joins`）で算出し命令に格納する。
    - **描画**: 立上り描画（`execute_walls`）が配置した壁ハンドルを**命令インデックスをキー**に dict へ記録し（横架材ハンドル→タグと同じ受け渡し方式）、`execute_wall_joins` が `a`／`b` で 2 壁のハンドルを引いて `vs.JoinWalls(first, second, pick_a, pick_b, join_type, capped, showAlerts=False)` を呼ぶ。ピック点は各壁の残す側に寄せた `pick_a`／`pick_b`（上記）を渡す。`capped` は命令ごとの値を渡す。`showAlerts=False` で結合失敗時のダイアログを抑止し、インポート中に手動操作を求められないようにする。どちらかの壁が未配置（レイヤ未生成・壁が作れずフォールバック描画でハンドル未記録）の命令はスキップする。実行順は立上り描画の直後（底盤の前）。`JoinWalls` の引数・挙動は VectorWorks 上で最終確認する（描画フェーズは他要素と同じく VW 上で検証する方針。`vw/footing.py` の `_JOIN_SHOW_ALERTS` に集約）。
- **底盤（基礎底盤・布基礎底盤・独立基礎底盤、`Name` に `底盤` を含む）→ スラブオブジェクト（slab 命令、`F-底盤` レイヤ）**。天端の絶対 Z を `elevation` に格納し、平面外形を底盤天端レベルにバインドする。`bound.offset` は実天端 Z と底盤天端の絶対 Z の差（主たる底盤は ≈0）。**底盤にはスラブスタイル用のコンクリート厚（`thickness`＝ソリッドの Z 厚を整数 mm に丸めた値）を持たせる**（下記スラブスタイル）。
  - **底盤のマージ（`merge_slab_commands`）**: ホームズ君 IFC では底盤が通り芯の升目等で細かく分断されているため、そのまま描くとスラブオブジェクトが多数に分かれる。**同じ厚さ・同じ高さ（同一グループ）で連続する底盤**を 1 枚のスラブに統合する。`build_slab_commands` は slab 命令を組み立てた後にこの統合を通す。断面キー（`_slab_merge_key`＝レイヤ・クラス・コンクリート厚・高さ基準の `story_offset`/`level`/`offset` を許容値 `_SLAB_MERGE_TOL`=1mm で丸めたもの）ごとにグループ化し、各グループ内で**辺を共有／面で重なる連続する底盤**（`_polys_connected`。角＝点だけで接するものは連続としない）の連結成分（`_slab_components`、Union-Find）を求め、成分ごとに**任意向きの単純多角形の和**（`_polygon_union`）を 1 枚の統合底盤にする。`_polygon_union` は各底盤を CCW に揃え、全辺を他辺との交点で細分し（`_seg_split_points`／`_split_edge`）、細分した有向辺のうち**辺のすぐ右（外側）がどの底盤にも含まれない**辺だけを境界として残し（共有辺は両隣の底盤が右側に来て打ち消され外周辺だけ残る）、残った辺をつないでループ化する（`_chain_boundary`。分岐点は最も時計回りの辺を選び内側を左に保つ）。軸平行の矩形だけでなく、**傾いた底盤（斜めの建物グリッド）や斜め辺＝45 度取合いの底盤も統合できる**（回転・矩形前提を持たないため）。**単独の底盤・和が穴を含む/複数外形に分かれる成分（＝布基礎の升目状ラティスなど。ベタで埋めると部屋の下までコンクリートになり誤りのため統合せず元のまま残す）・和の計算に失敗した成分（開ループ）・地中梁（`thickness=None`）は統合しない**。統合はグループ化・成分処理とも入力順に対して決定的。
  - **底盤外周の外面合わせ（`align_slabs_to_wall_faces`）**: ホームズ君 IFC の底盤外形は**立上り（基礎梁）の壁心**に一致している。実際の底盤外周は立上りの**外面**（壁心 + 半壁厚）まであるため、マージの後に底盤（`thickness` を持つスラブ）の外周を外面まで外側へ広げる。底盤の各辺に沿う立上りを探し（`_wall_half_thickness_for_edge`＝辺と壁芯が平行・同一直線上で区間が重なる立上り。最も重なりの大きいものを採る）、その**半壁厚**だけ辺を外向き法線方向へ平行移動し、隣接する移動後の辺の交点を新頂点にする（`_offset_polygon`。凸角は外へ伸び、入隅は詰まる可変量オフセット）。**立上りに沿う辺が 1 つも無い底盤（立上りの無い独立基礎底盤等）・地中梁は動かさない**。辺ごとに沿う立上りの厚みを使うため、外周で壁厚が混在しても各辺が正しい外面に合う。判定は入力順に対して決定的。外面合わせに使う立上りは `build_document` が組み立てた `walls` を `build_slab_commands(ifc_file, walls)` へ渡す（未指定なら内部で `build_wall_commands` を呼ぶ）。
- **地中梁（地中梁・部分地中梁、`Name` に `地中梁` を含む）→ スラブオブジェクト（slab 命令、`F-底盤` レイヤ）**。底盤の下にぶら下がるため実形状どおり底盤天端より低い天端にバインドする（`bound.offset` が負値）。これにより底盤スラブと噛み合う。**地中梁はスラブスタイルを適用しない**ため `thickness=None`。
- 描画は外形ポリゴンを作り `vs.CreateSlab`(profile)→（底盤のみ）スラブスタイルを適用→`vs.SetSlabHeight`(天端の絶対 Z=`elevation`)→`SetObjectStoryBound`(天端、`底盤天端`)。スラブが作れない場合は外形ポリゴンにフォールバック。**`SetSlabHeight` はスラブ厚ではなく天端高さ（Coordinate）を設定する関数**であり、以前は厚みを渡していたため天端が厚み分だけ高く描画されていた（柱・梁の高さ二重加算＝#54/#55 と同種の不具合、#70 で修正）。基礎ストーリは GL=0 なので `SetSlabHeight` に渡す天端の絶対 Z はストーリ基準高さと一致する。**スラブ厚（実体）はスラブスタイルのコンポーネントが決める**（下記）。
- **スラブスタイル（底盤、`vw/footing.py`）**: 底盤には**コンクリート厚に応じたスラブスタイル**（`基礎スラブ - コンクリート {厚}mm / 捨てコン …mm / 砕石 …mm`）を `vs.SetSlabStyle` で適用する。既定は 150mm 厚のスタイル `基礎スラブ - コンクリート 150mm / 捨てコン 30mm / 砕石 100mm`。**スタイルの一覧（名前→ハンドル）は `BuildResourceList`(種別 107=スラブスタイル、folderIndex=0=ドキュメント内)で列挙し、各ハンドルを `GetResourceFromList` で取得する**（`_slab_style_handles`）。**`vs.GetObject` はリソースマネージャのリソース（スラブスタイル）を確実には返さない**ため使わない（以前 `GetObject` で存在確認・複製元取得をしていたため、既定 150mm スタイルが NIL 扱いになりスタイルが一切適用されなかった不具合を修正）。**既定スタイルの探索は、捨てコン・砕石の既定厚が将来変わりうるため、コンクリート厚（150mm）だけを固定して残り（捨てコン・砕石の部分）を任意の文字列にマッチさせる**（`_find_base_slab_style`＝列挙した名前を ` / ` で 3 分割して先頭＝`基礎スラブ - コンクリート 150mm`・2 番目＝`捨てコン` 始まり・3 番目＝`砕石` 始まりのものを選ぶ）。**150mm 以外の厚みは既定スタイルのハンドルを複製し（`CreateDuplicateObject`。挿入先は `vs.GetParent` で得た親コンテナ＝リソースの複製に nil コンテナを渡すと無名の不正リソースを作りドキュメントを壊すため）、最上層（`#1`＝コンクリート）の厚みを `SetComponentWidth` で変更した新スタイルを作って適用する**（`_resolve_slab_style_ref`。新スタイル名は既定名のコンクリート厚部分だけを差し替えたもの＝捨てコン・砕石の表記は既定から引き継ぐ。例 180mm なら `基礎スラブ - コンクリート 180mm / 捨てコン 30mm / 砕石 100mm`）。作った新スタイルは一覧に登録するため、同一厚みの底盤が複数あっても複製・列挙は繰り返さない。**`SetSlabStyle` に渡すスタイルの ref 番号は正の `Name2Index(名前)`**（0＝解決不可のときはスタイルを付けない）。VW 上で確認済み: 負値（`-Name2Index`。線種等の名前付きリソースの規約）ではスラブスタイルは適用されず「スタイルなし」のまま、**正値でのみ適用**され `GetSlabStyle` も同じ正値を返す。既定スタイルが見つからない（複製元が無い）場合はスタイルを付けずに描画する。
- **構造クラス**: 立上り=`04構造-01基礎-03立ち上がり`、底盤・地中梁=`04構造-01基礎-02基礎スラブ`。`SetClass` で設定する（存在しないクラスは VW が自動生成）。

**底盤天端レベルの高さ**（`resolve_slab_top_elevation`）は、底盤（`底盤` を含む要素）の天端 Z ごとに平面面積を合計し、合計面積が最大の天端 Z を採用する（例: 大半の基礎底盤の天端 50.0、独立基礎底盤など少数の異なる高さは採用されない）。エンティティ列挙順に依存しない決定的な高さにするため最初に見つかった値ではなく面積最大の Z を使う。

**押し出しソリッドのワールド変換**: 底盤は鉛直押し出し（プロファイルがそのまま平面外形）だが、立上り・地中梁・布基礎底盤は水平押し出し（プロファイルが鉛直面内）。`_world_solid` が要素配置とアイテム配置（`IfcExtrudedAreaSolid.Position`）を合成した行列でワールド座標に変換し、押し出し方向の Z 成分が鉛直か水平かで平面外形の求め方を分ける（鉛直＝プロファイル多角形、水平＝断面の水平幅を押し出し方向に掃引した矩形）。端部が他材で削られた要素は `IfcBooleanResult`（差演算）で表されるため、第 1 オペランドの素のソリッドを辿る（削り分だけ長めになるが取り逃すよりは妥当）。非矩形断面の立上りは壁厚が定まらないためスキップする。

### 床板（ifc/floor.py → vw/floor.py）

ホームズ君 IFC の床板は `Name` が `床版` の `IfcSlab`（鉛直押し出しで、押し出しプロファイルがそのまま床の平面外形になる）として表現される。これを各階の `n-FL` レイヤに **床ツール（Floor オブジェクト）** で作図する。要件は「床のある場所は IFC から抽出・厚みは 24mm 固定・**IFC の床位置を尊重する（床の段差＝スキップフロアを表現する）**」。

- 解析（`ifc/floor.py` の `build_floor_commands`）: FL ストーリ（`Name` が `FL` で終わる `IfcBuildingStorey`）を Elevation 昇順に走査し、各階の `ContainsElements` から床版（`床版`）を集めてその階の FL レイヤ（`n-FL`）に配置する floor 命令にする。**最上階（屋根）は FL レイヤを持たない（軒高のみ）ため対象外**にする（床板は屋根には無い。フィクスチャでも床版は非最上階の FL 階にのみ現れる）。座標は通り芯・基礎と同じグリッド中心オフセットで補正する。平面外形の取得（`_world_solid`／`_footprint`）・最下端 Z の取得（`_z_top_and_thickness`）は基礎（`ifc/footing.py`）の低レベルヘルパーを再利用する（アンカーボルト・火打が private ヘルパーを再利用するのと同じパターン）。
  - **厚み**: 要件により **24mm 固定**（`FLOOR_THICKNESS`）。IFC の押し出し厚（実際には 28mm 等が出力されている）は使わない。
  - **高さ（IFC の床位置を尊重＝スキップフロアを表現）**: `elevation`（床下端の絶対 Z）を **IFC 床版ソリッドの最下端**（`storey.Elevation + ローカル最下端 Z`＝`_z_top_and_thickness` の `top − thickness`）そのままにする。高さ基準は**標準の床高＝横架材天端レベル**（`storey.Elevation + resolve_beam_top_offset(storey)`＝story.py が `横架材天端` レベルに使うのと同じ値）に `bound`（`story_offset=0`・`level=横架材天端`）でバインドし、`bound.offset` に**床下端と横架材天端の差分**（＝ホームズ君で入力した基準高さからの高低差）を格納する。段差の無い床は床下端＝横架材天端で `offset=0`、**段差のある床（スキップフロア）や床高の異なる床は `elevation`・`offset` がずれて表現される**（ホームズ君 IFC では、標準の床は床版のローカル配置 Z が横架材天端オフセットと一致し、段差床はそこから高低差ぶん下がった／上がったローカル Z を持つ。例＝スキップフロア_サンプルの 2FL は通常床が offset 0、段差床が offset −832）。以前は全床を横架材天端へ潰していたため段差が失われていた。
  - **クラス**: 床板＝`04構造-02木造-06耐力面材-02床`（`CLASS_FLOOR`。大引・根太・床束と同じ床組の階層に置く）。存在しないクラスは VW が自動生成する。
- 描画（`vw/floor.py` の `execute_floors`／`draw_floor`）: 配置先レイヤ（`n-FL`）をアクティブにし、床ツールで床を作る。**床ツールは `vs.BeginFloor(厚み)` で開始し、2D 図形（平面外形の閉じたポリゴン）を描いてから `vs.EndGroup()` で確定する**（`BeginFloor` の VW 公式ドキュメント: 「2D オブジェクト作成手続きでテンプレートを定義し、`EndGroup` で完了する」）。作成後、床下端を IFC の床位置（命令の `elevation`、絶対 Z）へ `vs.Move3D(0, 0, elevation)` で移動し（構造材の Move3D と同じ規約）、高さ基準を `vs.SetObjectStoryBound` で標準の床高＝`横架材天端` レベルに `bound.offset`（基準高さからの高低差）でバインドする（編集時に高さがずれないようにする。段差＝スキップフロアはこの `elevation`・`offset` に現れる）。床が作れない場合は外形ポリゴンにフォールバックする。配置先レイヤが存在しない命令はスキップする。実行順は底盤（スラブ）の後・配筋の前（`execute_document`）。**床の高さの与え方（Move3D の絶対 Z・厚みの伸びる向き・`SetObjectStoryBound` のアンカー）は他の要素と同じく VectorWorks 上で最終確認する方針**（床下端＝IFC の床位置になることを確認する）。床の内部プラグイン挙動（`BeginFloor`／`EndGroup`）は VW 公式スタブ（`vs.py`）と参照スクリプトで確認済み。

### 配筋（ifc/rebar.py → vw/rebar.py）

基礎の各要素に姉妹プロジェクト `vectorworks-plugin-rebar` のカスタム PIO **`鉄筋`**（3D パス図形。パスと配筋仕様＝モード・鉄筋から平面線・3D 鉄筋・断面 2D コンポーネントを自前で描くプラグインオブジェクト）を配置する。本リポジトリは PIO の配置とパラメータ設定までを担い、配筋の描画ロジックは持たない（下階柱記号 PIO と同じ方式）。配筋が編集されれば PIO のリセットで表現が追随する。**PIO は既存の基礎レイヤ（`F-立上り`／`F-底盤`）に重ねて配置する**（配筋の高さは PIO の 3D パスが持つ絶対 Z で決まり、レイヤのストーリレベルには依存しないため、専用レイヤを新設せず基礎伏図にそのまま載る）。

- 分類と配筋モード（`build_rebar_commands`）:
  - **立上り（基礎梁、`Name` が `基礎梁` 始まり）→ 梁モード**（`F-立上り` レイヤ）。断面（壁厚×壁高）の梁として、梁天端の中心線を 3D パスに、上下端筋・せん断補強筋を配筋仕様にする。**梁モードは 1 要素につき 1 本の直線パス（1 本 1 直線）**にする（立上り・地中梁とも単一方向の押し出しなので中心線は 2 点の直線。鉄筋 PIO は折れ線の梁だと断面ビューポートで横断面と側面図が両方表示されるため 1 本 1 直線でモデル化する運用。同一直線上の立上りを 1 本にまとめず要素ごとに置くのはこの運用に沿い、かつ要素ごとに `Pset_Reinforcement` が異なりうるため）。
  - **地中梁（地中梁・部分地中梁、`Name` に `地中梁` を含む）→ 梁モード**（`F-底盤` レイヤ＝地中梁をスラブとして描くレイヤに重ねる）。立上りと同じ扱いだが、断面が台形等の**任意形状**のため、断面のバウンディングボックス（水平方向の幅×鉛直方向の高さ）を梁断面の近似とし、中心線は断面の水平中心（プロファイルの u 範囲の中点＝`lx` 方向）を通す。矩形断面の立上りでは u 範囲が対称なので中心線は配置原点を通り幅=壁厚・高さ=壁高になる（`_beam_geometry` は両者を統一的に扱う）。
  - **底盤（基礎底盤・布基礎底盤・独立基礎底盤、`Name` に `底盤` を含む）→ スラブモード**（`F-底盤` レイヤ）。底盤天端の外形を 3D パス（閉じた多角形）に、主筋・配力筋（シングル・両方向）とスラブ厚を配筋仕様にする。
- **配筋仕様の取得**: IFC の `Pset_Reinforcement` プロパティセット（`IsDefinedBy` 経由）から取得する。立上り・地中梁は `TopReinforce`／`BottomReinforce`（例 `SD295_1-D13` / `2-D13`）・`ShearReinforce`（例 `SD295_1-D10@300`）、底盤は `TopReinforce`（例 `SD295_D13@300_D13@300`＝2 方向、`SD295-D13@200`＝1 方向）。鋼種の接頭辞（`SD295_` / `SD295-`＝`_GRADE_RE`）を除き、せん断補強筋は本数の接頭辞（`1-`）も除いて PIO の仕様文字列（`120×500` / `1-D13` / `D10@300` / `D13@300`）にする。**プロパティが無い（IFC に配筋情報が出力されていない）要素には既定値**（`_beam_bars`／`_slab_bars`＝立上り・地中梁は上下 1-D13・せん断 D10@250、底盤は D13@150 シングル両方向）を使う。
- **座標・ジオメトリ**: 通り芯・基礎と同じグリッド中心オフセットで補正する。押し出しソリッドのワールド変換・断面外形の取得は `footing.py` の低レベルヘルパー（`_world_solid`／`_z_top_and_thickness`／`_footprint`／分類判定）を再利用する。梁モードのパス Z は断面天端（始終点で一致）、スラブモードのパス Z は底盤天端。PIO はこの Z を天端として断面を下方向に取る。
- **クラス**: PIO 本体の描画クラス（PIO は全図形をこのクラスで描く）は `04構造-01基礎-09鉄筋`（`CLASS_REBAR`）。立上り・地中梁・底盤の配筋を 1 つのクラスにまとめ、コンクリートの構造クラスとは分けて配筋の線種・色を独立して調整できるようにする。
- 描画（`vw/rebar.py` の `execute_rebars` / `draw_rebar`）: 配置先レイヤ（`F-立上り`／`F-底盤`）をアクティブにし、命令の 3D パス頂点から `vs.Poly3D`（梁モード=開いた線＝`OpenPoly`、スラブモード=閉じた多角形＝`ClosePoly`）で 3D ポリゴンを作り、PIO を配置。**作成時のダイアログ抑止**: `vs.CreateCustomObjectPath` で 3D パス図形を作ると、プラグイン設定によっては作成のたびに「オブジェクトの設定」ダイアログが開いてインポートが中断する（`CreateCustomObjectPath` には点オブジェクトの `showPref` に相当する引数が無い）。これを避けるため、点オブジェクト（`柱束伏図記号`）と同じ **`vs.CreateCustomObjectN('鉄筋', (0,0), 0, showPref=False)`（ダイアログ抑止）で PIO を原点に作り、`vs.SetCustomObjectPath(obj, path)` で 3D パスを後付けする**（VectorWorks 公式ドキュメントのパス図形スクリプト作成パターン）。`SetCustomObjectPath` は**パスを無変換で使用する**ため、原点＝恒等配置の PIO にワールド座標のパスをそのまま与えれば配置は正しくなる（`CreateCustomObjectPath` は先頭頂点を原点へ平行移動しオブジェクト配置に絶対位置を持たせるが、最終的なワールド位置は同じ）。`CreateCustomObjectN` が NIL を返す環境では `CreateCustomObjectPath` にフォールバックする（作成時ダイアログが出る場合はあるが配筋の配置は行う）。PIO 本体を `vs.SetClass` で命令の `class` に設定し描画属性をすべてクラス属性に従わせてから、モード（`Mode`＝`梁`／`スラブ`）と配筋仕様を `vs.SetRField` で PIO のパラメータ（`SectionSize`／`TopBars`／`BottomBars`／`Stirrup`＝梁、`MainBar`／`DistBar`／`SlabThickness`＝スラブ）に設定し、`vs.ResetObject` でリセットする。**リセット時に PIO 本体（=vectorworks-plugin-rebar のコード）がパス・パラメータから配筋を描く**。配置先レイヤが存在しない命令・PIO が作れない（プラグイン未登録で `CreateCustomObjectPath` が NIL）命令はスキップする。実行順は底盤の後・アンカーボルトの前（`execute_document`）。PIO 名 `鉄筋`・パラメータ名は VectorWorks 側のプラグイン登録名（および vectorworks-plugin-rebar の `vw/pio.py` の `PARAM_*`）と一致させる必要がある（`vw/rebar.py` 冒頭の名前付き定数に集約）。

### アンカーボルト（ifc/anchor_bolt.py → vw/anchor_bolt.py）

ホームズ君 IFC ではアンカーボルトは `IfcMechanicalFastener` として表現され、型（`IfcMechanicalFastenerType`）の名前がボルト本体か座金かを区別する。ボルト本体を **ハイブリッドシンボル** に置換して基礎ストーリの `F-アンカーボルト` レイヤに配置する。

- 解析: 型名が `アンカーボルト:` で始まる（＝ボルト本体。角座金 `アンカーボルト座金:…` はこの接頭辞に一致しないため除外される）`IfcMechanicalFastener` を対象にする。**各アンカーボルトはボルト本体・座金の 2 要素で表される**ため、ボルト本体だけを採る（座金は同一軸芯にあり両方を数えると二重になる。座金なしのボルトは座金要素が無いため、ボルト本体を採ればすべてのアンカーボルトを漏れなく数えられる）。型は柱頭・柱脚金物と同じく `IfcRelDefinesByType` 経由で辿る（`ifc/column.py` の `_get_fastener_type_name` を再利用。逆方向属性名がスキーマで異なり IFC2X3=`IsDefinedBy`／IFC4=`IsTypedBy`）。
  - **シンボル判定（M12/M16）**: ホームズ君 IFC のサンプルには M12/M16 の直接表記が無く、代わりに Z1/Z2 の種別コード（角座金 Z1=40×40、Z2=60×60）と `座金なし` がある。要件により **座金付き（型名が `座金なし` を含まない）→ `アンカーボルト_M12`、座金なし（型名が `座金なし` を含む）→ `アンカーボルト_M16`** に振り分ける（`resolve_anchor_bolt_symbol`）。ボルト径のモデル値は種別に依らず一定（半径 10mm）で判定に使えないため、型名の `座金なし` トークンだけで判定する。
  - **2D 基準位置**: アンカーボルトの軸芯（ローカル配置 `Location` の XY を通り芯と同じグリッド中心オフセットで補正。`ifc/column.py` の `_get_position_2d` を再利用）。アンカーボルトは `設計GL` ストーリに含まれるが XY 原点は他要素と共通のため同じ補正でよい。
  - **高さの基準**: 基礎天端。命令には高さ情報を持たせず、配置先レイヤ `F-アンカーボルト` のストーリレベル（基礎天端）が担う（下記ストーリ節参照）。
- 描画: `vs.Symbol(シンボル名, (x, y), 0)` でハイブリッドシンボルを軸芯位置に配置する。アクティブレイヤ（`F-アンカーボルト`）が基礎天端レベルにバインドされているため、シンボルはそのレイヤ平面（＝基礎天端）に置かれる。配置先レイヤが存在しない命令はスキップする。
- **基礎天端の高さ**（`ifc/footing.py` の `resolve_foundation_top_elevation`）は立上り（基礎梁）天端 Z の最大値。立上りが無い基礎（底盤のみ）は底盤天端にフォールバックする。この値を基礎ストーリの `基礎天端` レベル（`F-アンカーボルト` レイヤ）のオフセット（絶対 Z）に用いる。

### 床束（ifc/floor_post.py → vw/floor_post.py）

ホームズ君 EX の IFC には**床束が一切出力されない**（オブジェクト・型・プロパティのいずれにも床束・束の位置や仕様が現れない。全フィクスチャで確認済み）。そのため床束は IFC から抽出できず、要件どおり**大引の下に 910mm 間隔で決め打ち配置**する。床束を **ハイブリッドシンボル** `床束` に置換して基礎ストーリの専用レイヤ `F-床束` に配置する。

- 解析（`build_floor_post_commands`）: `Name` の種別が `大引`（`member_class_from_name` が `CLASS_OOBIKI` を返す）の `IfcBeam` / `IfcMember` を対象にする。座標は通り芯・横架材と同じグリッド中心オフセットで補正する。ジオメトリの取得は横架材（`ifc/member.py`）の `_get_placement_3d`・`_get_profile_dims` を再利用する（アンカーボルトが `ifc/column.py` の private ヘルパーを再利用するのと同じパターン）。
  - **継手の統合（`_merge_collinear_ohbiki`）**: ホームズ君 IFC の大引は継手（支持材の上での継目）で分断されている（1 連が `n×910−105mm` の実部材ごとに分かれ、継目に支持材幅ぶん＝約 105mm のすき間ができる）。継手は実際の大引の端ではないため、床束の間隔計算では**同一直線上で継手（すき間 ≤ 半モジュール＝`_JOINT_GAP_TOL`）で分断された大引を 1 連に統合**してから配置を計算する（継手は無いものとして 910mm 間隔を連の全長に通しで割り付ける）。Union-Find で同一直線上（`_collinear_gap`＝方向が平行かつ相手端が芯線上、`_COLLINEAR_ANGLE_TOL`／`_COLLINEAR_PERP_TOL`）ですき間許容内の大引をまとめ、各成分を先頭の芯線方向へ全端点を射影した最小〜最大区間の 1 本にする。継手のすき間（支持材幅≈105mm）は半モジュール（455mm）を大きく下回り、別々の大引の間隔（≥1 モジュール≈1000mm、フィクスチャで確認済み）を大きく下回るため、この閾値で継手と別材を切り分けられる（**同一直線上でも 1 モジュール以上離れた別々の大引は統合しない**＝存在しない大引の下に床束を置かない）。統合は入力順に依存しない。以降の端部・配置間隔は統合後の 1 連に対して行う。
  - **端部＝支持材芯（`_shin_reference`）**: 間隔の基準となる「大引端部」は**実部材端ではなく、その端を受けている支持材（土台または他の大引）の芯（支持材芯）**とする。ホームズ君 IFC の大引は端部が支持材芯より半支持材厚だけ内側に納まって描かれており（1 モジュール＝910mm 区間で実長 805mm＝910−105）、実部材端を基準にすると床束が実際より内側（支持材芯 1 個分の半端）に寄る。支持材の候補は土台（`CLASS_DODAI`）と大引（`CLASS_OOBIKI`）の芯線（`_collect_support_lines`、`_SUPPORT_CLASSES`）。各端で大引の芯線と交わる支持材の芯線の交点を求め、その交点が支持材の区間内かつ大引端から半支持材厚（＋`_SHIN_MARGIN`）以内にある（＝大引端がその支持材の footprint に載る）支持材のうち最も近い交点を支持材芯とする。**他の大引の上に載る二次大引の端も、受けている大引の芯を端部にする**。自身の芯線・同一直線上の大引は平行のため（`_PARALLEL_TOL`）除外され、どの支持材にも受けられていない端（基礎に直接載る端等）は実部材端にフォールバックする。支持材芯は `build_wall_commands` 等と同じくグリッド中心オフセット前の生座標で求め、床束位置に反映する際にオフセットを引く。
  - **配置間隔（`_post_offsets`）**: 統合後の大引 1 連について、支持材芯どうしの区間に沿って**始点側の支持材芯（端部）から 910mm ずつ**（`910mm`・`1820mm`・…）床束を置く。最後の床束と終点側の支持材芯（反対側の端部）との間隔は**910mm 未満の半端になってよい**（`_POST_INTERVAL * k < length` を満たす `k` の位置に置く。`length`＝支持材芯区間長）。支持材芯そのものには床束を置かない（端部は支持材が受ける）。継手統合により支持材芯区間は連の全長（両端が支持材に受けられる場合はおおむね `n×910mm`）になるため、**支持材芯区間が 910mm 以下の連は床束 0 本**、`n×910mm` の連は `n−1` 本になる。判定は幾何のみに依存し入力順に依存しない。
  - **高さの基準**: 基礎底盤上端（底盤天端）。命令には高さ情報を持たせず、配置先レイヤ `F-床束` のストーリレベル（`床束`＝底盤天端に揃える。上記ストーリ節参照）が担う。
  - **基礎が無いモデル**: 床束は基礎（底盤）の上に立つため、`has_foundation` が False（配置先レイヤ `F-床束` が生成されず高さ基準も定まらない）の場合は空リストを返す。
- 描画（`vw/floor_post.py` の `execute_floor_posts` / `draw_floor_post`）: `vs.Symbol('床束', (x, y), 0)` でハイブリッドシンボルを配置する。アクティブレイヤ（`F-床束`）が床束レベル（＝底盤上端）にバインドされているため、シンボルはそのレイヤ平面（＝底盤上端）に置かれる。配置先レイヤが存在しない命令はスキップする（アンカーボルトと同じ扱い）。実行順はアンカーボルトの後・火打の前（`execute_document`）。
- **伏図への表示**: `F-床束` レイヤは基礎伏図ビューポートの表示レイヤに加える（`ifc/sheet.py`、下記シート節。底盤・立上りの上に重ねて表示する）。**シンボル名 `床束`・配置間隔 910mm は要件で決め打ちした値**であり、`ifc/floor_post.py` 冒頭の名前付き定数（`SYMBOL_FLOOR_POST`・`_POST_INTERVAL`）に集約している。`床束` シンボルは VectorWorks 側で登録したハイブリッドシンボル名と一致させる必要がある。

### 火打（ifc/fire_brace.py → vw/fire_brace.py）

ホームズ君 IFC では火打（火打梁）は `Name` が `火打:…`（例 `火打:1_1`）の `IfcBeam` / `IfcMember` として表現される。押し出し方向が鉛直（`Axis=(0,0,1)`）で断面が `IfcArbitraryClosedProfileDef`（平面外形が火打の footprint）のため、横架材（矩形断面の水平押し出しを扱う `ifc/member.py`）では拾われない（鉛直軸・非矩形断面のいずれの条件でもスキップされる）ので専用に処理する。火打を **ハイブリッドシンボル** `鋼製火打` に置換し、火打が属する階の横架材レイヤ（一般階 `n-横架材天端`・最上階 `n-軒高`＝横架材と同じレイヤ）に配置する。

- 解析（`build_fire_brace_commands`）: 横架材と同じく FL ストーリを走査し、各ストーリの `ContainsElements` から `Name` が `火打` 始まりの `IfcBeam`/`IfcMember` を対象にする。配置先レイヤは属する階の横架材レイヤ。座標は通り芯・横架材と同じグリッド中心オフセットで補正する。
  - **基準点（シンボルの挿入位置）＝横架材接合部の内側面交点**: 火打は直交する 2 本の横架材の間に斜めに架かり、両端の面がそれぞれの梁の側面に取り付く。火打の平面外形（footprint）は 2 本の長辺（材の長さ方向に平行）と 2 つの端面（各梁に取り付く面）からなる。**2 つの端面の直線を延長した交点**が「梁に取り付く 2 面が幾何学的に交わる内角の点」になる（`_base_point`）。footprint の取得は基礎と同じ `ifc/footing.py` の `_world_solid`（要素配置＋アイテム配置を合成しプロファイル頂点をワールド化）を再利用する。端面は、プロファイルが中心線（局所 Y=0）に対称なことを利用し、**局所 Y の符号が始終点で反転する（中心線をまたぐ）辺**として識別する（長辺は局所 Y が一定）。端面がちょうど 2 つ得られない・平行で交点が求まらない火打はスキップする。
  - **回転角（`angle`）**: ユーザー指定により火打の向きに合わせて回転する。基準点（内角）から火打本体の重心へ向かう方向（＝内角の二等分方向）に、シンボルの基準姿勢のずれを補正する反時計方向 45 度（`_SYMBOL_ANGLE_OFFSET`）を加える（`_angle`）。4 方向の直交コーナーだけでなく非 90 度コーナー（斜め壁等）にも追従する。この補正値は `鋼製火打` シンボルの基準姿勢（0 度での向き）を VectorWorks 上で確認した結果に基づく。
  - **高さの基準**: 横架材天端（最上階は軒高）。命令には高さ情報を持たせず、配置先レイヤのストーリレベルが担う（横架材と同じ扱い）。
- 描画（`vw/fire_brace.py` の `execute_fire_braces` / `draw_fire_brace`）: `vs.Symbol('鋼製火打', (x, y), angle)` でハイブリッドシンボルを基準点に配置する。アクティブレイヤ（横架材天端／軒高）がストーリレベルにバインドされているため、シンボルはそのレイヤ平面に置かれる。配置先レイヤが存在しない命令はスキップする。実行順はアンカーボルトの後・シートの前（`execute_document`）。

### 仕口（ifc/joint.py → vw/joint.py）

**受ける材のある横架材端部**にハイブリッドシンボル `仕口` を配置する。横架材の端部が別の横架材（＝受ける材）に取り付く場合、および**柱の側面に取り付く場合**にだけ、その端部にシンボルを置く。**基準点は梁端の中央上端**とし、**横架材と同じレイヤ**（一般階 `n-横架材天端`・最上階 `n-軒高`・母屋 `n-母屋`）に描画する。**他の部材に取り付かない自由端には配置しない**。IFC には仕口が出力されないため、横架材の member 命令（食い込み調整済み）・柱の column 命令の平面ジオメトリから受ける材の有無を導出する（火打・タグと同じく IFC を直接見ず member 命令・column 命令を再利用する。`build_document` は横架材命令・柱命令を一度だけ組み立てて `members`・`columns` と `build_joint_commands(members, columns)` に渡す）。置換シンボル名 `仕口` は `ifc/joint.py` 冒頭の名前付き定数（`SYMBOL_JOINT`）に集約している。

- 解析（`build_joint_commands`）: 横架材 1 本の各端部（始端・終端）について、**受ける材（別の横架材）または柱に取り付くか**を判定し、取り付く端部にだけ joint 命令を出す。座標は member 命令の座標をそのまま使う（既にグリッド中心オフセット済み）。柱を渡さない（`columns=None`）場合は従来どおり横架材同士だけで判定する。
  - **受ける横架材の判定（`_end_has_receiver` / `_point_in_member`）**: ある端点が、**同一レイヤ・Z 範囲が重なる・軸が交差する（平行でない）別の横架材の footprint（矩形）に入る**（内部・面上に載る）とき、その相手を受ける材とみなす。相手材の中心線からの軸方向位置 t が `[0, 相手長]`（コーナー＝端も含める余裕 `_ALONG_TOL`）にあり、直交距離が半幅 + `_FACE_TOL`（面ちょうどに載る端部も含む）以内なら取り付き。T 字（相手の途中に突き当たる）・L 字（コーナーで相互に食い込む）のどちらも受ける材ありと判定する。member 命令は食い込み調整（`resolve_member_interferences`）済みで、負け側の端部は相手の面ちょうどまで詰められているため、面に載る端部を取り付きとみなすことで T 字の受ける側を正しく検出する。**平行な相手（同一直線上の継ぎ手・側並びの平行材）は受ける材にしない**（軸の外積が `_PARALLEL_TOL` 未満）。**別レイヤ・Z 範囲が離れた段差材も受ける材にしない**。この方針は横架材の食い込み調整（`ifc/member.py`）とそろえている（同一レイヤどうしの横架材だけを取り合いの相手にする）。判定は member 命令のジオメトリに対して行い**命令の並び順に依存しない決定的な結果**になる。
  - **受ける柱の判定（`_column_geom` / `_point_in_column`）**: ある端点が、**Z 範囲が重なる柱の断面矩形（幅×成・配置座標中心・軸平行）に入る**（内部・面上に載る）とき、その柱を受ける材とみなす。柱は横架材とは別レイヤ（`n-柱`）に置かれ方向を持たないため、レイヤ一致・平行判定はせず、断面矩形への内包（各軸で半幅/半成 + `_FACE_TOL` 以内）と Z 重なりだけで判定する。柱の Z 範囲は下端（`elevation`）〜上端（`elevation + height`）で、**梁の Z 範囲に柱が掛かる場合**（梁が柱の側面に取り付く＝直下階の管柱の上端・通し柱の途中）に取り付きとみなす。梁が上に乗るだけで柱の下端面に触れるだけの柱（当階の柱＝梁の上に立つ柱）は Z が面で触れるだけで重なりが `_Z_OVERLAP_TOL` を超えず対象外になる。判定は column 命令のジオメトリに対して行い命令の並び順に依存しない。
  - **基準点（`position`）**: 梁端の中央上端。member 命令の `start`／`end` は断面の左右中央・上端（天端中央）が通る線の端点なので、その端点をそのまま基準点にする。
  - **回転角（`angle`）**: 梁端の向きに合わせ、梁軸に沿って**端部から部材内側へ向かう方向**（始端は始端→終端、終端は終端→始端）にする（シンボルが梁と揃うようにする）。シンボルの基準姿勢は VectorWorks 上で最終確認する（描画フェーズは他要素と同じく VW 上で検証する方針）。
  - **高さの基準**: 横架材天端（最上階は軒高、母屋は母屋）。命令には高さ情報を持たせず、配置先レイヤのストーリレベルが担う（火打と同じ扱い）。
- 描画（`vw/joint.py` の `execute_joints` / `draw_joint`）: `vs.Symbol('仕口', (x, y), angle)` でハイブリッドシンボルを基準点に配置する。アクティブレイヤ（横架材天端／軒高／母屋）がストーリレベルにバインドされているため、シンボルはそのレイヤ平面に置かれる。配置先レイヤが存在しない命令はスキップする（火打と同じ扱い）。実行順は火打の後・下階柱記号の前（`execute_document`）。**`仕口` シンボルは VectorWorks 側で登録したハイブリッドシンボル名と一致させる必要がある**。

### 断面記号（ifc/column_mark.py → vw/column_mark.py）

姉妹プロジェクト `vectorworks-plugin-column-under-mark` のカスタム PIO **`柱束伏図記号`**（指定レイヤ・クラスの構造用途 4/5 の構造材を検索し、記号スタイル＝`MarkStyle` に従って記号を各位置に描くポイントオブジェクト。**断面記号は柱＝×・小屋束＝/＝実断面の外接矩形に合わせた対角線**）を配置する。柱が編集されれば PIO のリセットで記号の位置・数が追随するため、伏図と構造材の食い違いを防げる。命令（`ColumnMarkCommand`）は記号スタイル `style`（`断面`＝PIO の `MarkStyle` に渡す）を持つ。

柱・小屋束は span（またぐレベル区間）ごとの専用レイヤ `{from}to{to}-柱` に配置されるため（柱節・ストーリ節参照）、**実在する span レイヤごとに断面記号 PIO を 1 つ重ねる**。配置レイヤ・検索対象レイヤをともにその span レイヤ自身にし、`target_class` は空（全クラス）にして構造用途 4/5 の構造材すべてを記号化する。`style=断面`（`MARK_STYLE_SECTION`）で PIO の `MarkStyle` に渡し、**作図クラス `class` は極細実線 `01作図-01線-02実線-01極細線`（`SECTION_MARK_CLASS`）**。各伏図はその span レイヤを表示すると柱・小屋束と併せて断面記号も自動的に載る（span レイヤの表示可否はシートが切断レベルで決める）。

- 解析（`build_column_mark_commands(columns)`）: `collect_column_spans(columns)` で実在する span レイヤ（`(from, to)` 昇順）を列挙し、各レイヤに断面記号 1 つを組み立てる（配置レイヤ・検索対象レイヤともに span レイヤ、`target_class` は空、`style=断面`、作図クラス `SECTION_MARK_CLASS`）。IFC ジオメトリは参照せず柱の span から決まる。`size` は既定 300mm（`DEFAULT_MARK_SIZE`）、`position` は原点（記号は検索した柱のワールド位置に描かれ挿入点に依存しない）。
- 描画（`execute_column_marks` / `draw_column_mark`）: 配置先の span レイヤをアクティブにし、`vs.CreateCustomObjectN('柱束伏図記号', (x, y), 0, False)`（`showPref=False` でインポート中の設定ダイアログを抑止）で PIO を配置。**PIO 本体を `vs.SetClass` で命令の `class` に設定してから**、検索対象レイヤ・クラス・記号サイズ・記号スタイルを `vs.SetRField` で PIO のパラメータ（`TargetLayer`／`TargetClass`／`MarkSize`／`MarkStyle`＝姉妹プロジェクトの `PARAM_*` と一致）に設定し `vs.ResetObject` でリセットする（リセット時に PIO 本体が対象レイヤの構造材を検索して記号を描く）。配置先レイヤが無い命令・PIO が作れない命令はスキップ。実行順は火打の後・シートの前（`execute_document`）で、柱の描画完了後。
- **母屋伏図への小屋束の平面記号（伏図記号）、および従来の下階柱記号**: span 方式では柱の表示自体をレイヤ切断レベルで制御する（下階柱＝通し柱は跨ぐ階の床伏図に span レイヤとして自然に出る）ため、平面の伏図記号は役割が変わる。母屋伏図向けの小屋束伏図記号は span 方式への移行後に別途対応する予定で、現状この節は断面記号のみを扱う。PIO 名 `柱束伏図記号`・パラメータ名（`TargetLayer`／`TargetClass`／`MarkSize`／`MarkStyle`）は VectorWorks 側のプラグイン登録名と一致させる（`vw/column_mark.py` 冒頭の名前付き定数に集約）。

### シートレイヤ・伏図（ifc/sheet.py → vw/sheet.py）

モデルデータ（ストーリ・通り芯・基礎など）を取り込んだ後に、特定のデザインレイヤ群を表示するビューポートを配置した**シートレイヤ**を作成する。IFC からシート構成を読み取るのではなく、取り込んだ要素の有無からどのシートを作るべきかを解析フェーズが判断する。現状は **基礎伏図**（`build_foundation_sheet_commands`）・**各階の柱梁伏図**（`build_floor_framing_sheet_commands`）・**屋根版を持つ階ごとの母屋伏図**（`build_moya_sheet_commands`）を作成する。`build_sheet_commands` はこの順に並べた sheet 命令リストを返す（シートレイヤ番号 `1`＝基礎伏図、`2`〜＝各階の柱梁伏図、その後＝屋根版を持つ階ごとの母屋伏図を Elevation 昇順で）。

- 解析・基礎伏図（`ifc/sheet.py` の `build_foundation_sheet_commands`）: **基礎要素が存在する場合にだけ**基礎伏図の sheet 命令を返す（`footing.has_foundation` で判定。基礎が無ければ表示すべき `F-底盤`・`F-立上り`・`F-床束`・`F-アンカーボルト` レイヤが生成されずビューポートが空になるため、空リストを返す）。sheet 命令は シートレイヤ番号 `1`・タイトル `基礎伏図`、ビューポートの図面タイトル `基礎伏図`・図番 `1`、表示レイヤ `F-底盤`・`F-立上り`・`F-床束`・`F-アンカーボルト`・`共通`（底盤・立上り・床束・アンカーボルト・通り芯の順）を持つ。加えて**非表示クラス `hidden_classes`＝配筋クラス `04構造-01基礎-09鉄筋`（`FOUNDATION_PLAN_HIDDEN_CLASSES`）**を持つ。配筋（鉄筋 PIO）は立上り・底盤と同じレイヤ（`F-立上り`／`F-底盤`）に重ねて配置されるためレイヤでは絞れず、基礎伏図では配筋を隠して断面でのみ表示する要件のため、ビューポート単位で配筋クラスを非表示にする。
- 解析・柱梁伏図（`ifc/sheet.py` の `build_floor_framing_sheet_commands`）: **ストーリ（`collect_stories`）1 つにつき伏図 1 枚**。シートレイヤ番号・図番は基礎伏図（`1`）に続けて `2` から順（最下階＝`2`）。タイトル・図面タイトルは最下階から `1階床伏図`・`2階床伏図`・…、最上階は主屋根が架かる階番号を付けた `{階数}階小屋伏図`（`floor_plan_title`。例 2 階建てなら `2階小屋伏図`）。**下屋根（下屋）の小屋組（母屋・垂木・野地板）は柱梁伏図には重ねず、専用の母屋伏図（下記）に分けて表示する**（母屋等の小屋組と梁組が重なって見にくくなるのを防ぐため。タイトルにも母屋の表記は付けない）。表示レイヤは各階の**横架材**（一般階 `n-横架材天端`・最上階 `n-軒高`）・**通り芯**（`共通`）を基本とし、**その伏図の切断レベルを span が含む柱レイヤ**（`{from}to{to}-柱`。`_span_layers_at_cut`）を表示する。**切断レベルはその階の床レベル（1 始まり＝index+1）+0.25**（`FLOOR_PLAN_CUT_OFFSET`。1 階床伏図＝1.25・2 階床伏図＝2.25・2 階小屋伏図＝3.25）。span [from, to] が切断レベルを含む（`from ≤ 切断 ≤ to`）柱だけを載せることで、①その階を base とする柱の断面（`from=`その床レベル）と、②下から貫いてこの高さに達する通し柱（例 `1to3` は 1.25 と 2.25 の両方を跨ぐので両階に出る）が表示され、③**下屋の小屋束（`2to2.5`、to=2.5＜3.25）は上階の小屋伏図に写り込まない**（＝本改修の解消点）。加えて**最上階以外は床**（`n-FL`）を、**最下階（1 階）には基礎がある場合のみアンカーボルト**（`F-アンカーボルト`）も表示する。**下屋根（下屋）の小屋組（母屋・垂木・野地板）は柱梁伏図には重ねず、専用の母屋伏図（下記）に分けて表示する**（タイトルにも母屋の表記は付けない）。レイヤの列挙順は描画順＝重ね順には影響せず、重ね順はドキュメントのデザインレイヤ順が担う（床＝`n-FL` はストーリ節の `reorder_story_layers` で全ストーリ分をまとめて最背面へ回すため柱梁を隠さない）。span レイヤを切断レベルで絞るため column 命令（`columns`）を渡す（未指定なら内部で組み立てる）。ストーリが無ければ空リストを返す。
- 解析・母屋伏図（`ifc/sheet.py` の `build_moya_sheet_commands`）: **屋根版（屋根面）を持つ階ごとに、その階の小屋組（母屋・垂木・野地板）を梁組と分けて表示する伏図を 1 枚**作る。最上階（屋根）の主屋根だけでなく、中間階に架かる下屋根（下屋）にも 1 枚ずつ作る（`collect_story_roof_flags` で判定）。シートレイヤ番号・図番は各階の柱梁伏図の最後に続けて Elevation 昇順（最下階→最上階）に振る（基礎伏図 `1` ＋各階柱梁伏図＝ストーリ数、の次から）。タイトル・図面タイトルは屋根が架かる階番号（`moya_plan_title`＝ストーリ index を 0 起点でそのまま用いた `{index}階母屋伏図`。例 2 階建てなら主屋根＝`2階母屋伏図`、下屋根＝`1階母屋伏図`）。表示レイヤは 母屋（`n-母屋`、母屋がある階のみ）・垂木（`n-垂木`）・野地板（`n-野地板`）・通り芯（`共通`）に加え、**その伏図の切断レベルを span が含む柱レイヤ**（`{from}to{to}-柱`。`_span_layers_at_cut`）。**切断レベルはその階の床レベル（1 始まり＝index+1）+0.75**（`MOYA_PLAN_CUT_OFFSET`。1 階母屋伏図＝2.75・2 階母屋伏図＝3.75）。span [from, to] が切断レベルを含む（`from ≤ 切断 ≤ to`）柱だけを載せることで、**屋根を貫いて立ち上がる主屋の柱（管柱・通し柱、例 `2to3`・`1to3`）**が表示される。母屋伏図は母屋の上からの見下げ図なので小屋束の断面は出さず、母屋を支える下屋の小屋束（span の to が半整数で切断より低い、例 `2to2.5`＝to=2.5＜2.75）はこの切断に含まれず載らない（小屋束の位置は平面の伏図記号で示す＝span 方式への移行後に別途対応する）。柱の span レイヤを切断レベルで絞るため column 命令（`columns`）を渡す（未指定なら内部で組み立てる）。下屋根は母屋を持たない（単純な片流れ等）こともあるため、母屋レイヤは母屋がある階（`collect_story_moya_flags`）にだけ加え、垂木・野地板は屋根版がある階に加える。ストーリが無ければ空リストを返す。母屋・棟木を `n-母屋` レイヤに分離しているため（横架材節参照）、母屋・棟木のデータタグもその母屋レイヤを表示する母屋伏図にだけ載る（タグの `layer` がビューポートの表示レイヤに含まれる伏図に載るため。断面寸法データタグ節参照）。
- 描画（`vw/sheet.py` の `execute_sheets`）: **シートレイヤ番号はレイヤ名が担う**ため、シートレイヤを `vs.CreateLayer(number, 2)`（番号を名前に、種別 2 = プレゼンテーション＝シートレイヤ）で作成し（同番号があれば再利用）、シートレイヤタイトルを `vs.SetObjectVariableString`（selector `159`）で設定する。その上に `vs.CreateVP(sheetLayer)` でビューポートを作り、**表示レイヤを絞り込む**（`configure_viewport_layers`: 全デザインレイヤを `FLayer`→`NextLayer` で辿ってビューポートの親シートレイヤ以外を一旦非表示にし＝`SetVPLayerVisibility(vp, layer, 1)`、命令の `layers` に挙げたレイヤだけを名前で引いて表示に戻す＝`SetVPLayerVisibility(vp, layer, 0)`）。**`SetVPLayerVisibility` の表示種別は `0`=表示・`1`=非表示・`2`=グレーで、対象外レイヤに `2` を渡すとグレー表示で残ってしまうため `1`（非表示）で完全に隠す**。**クラスは既定で全表示、ただし命令の `hidden_classes` に挙げたクラスだけ非表示にする**（`configure_viewport_classes`: ビューポートは既定で一部クラスが非表示になるため、`ClassNum`／`ClassList` で全クラスを辿り、`hidden_classes` に含まれるクラスは `SetVPClassVisibility(vp, class, 1)`＝非表示、それ以外は `0`＝表示に設定する。伏図に必要な要素が欠けないよう、レイヤと違いクラスは基本的に絞り込まないが、表示レイヤに重なる配筋のようにクラスでしか分けられない要素だけをビューポート単位で隠す。基礎伏図は配筋クラスを隠す）。**縮尺は表示するデザインレイヤに合わせる**（`configure_viewport_scale`: 表示レイヤのうち最初に見つかったデザインレイヤの縮尺を `GetLScale` で取り、`SetObjectVariableReal(vp, 1003, scale)` でビューポート縮尺に設定する。伏図では表示レイヤの縮尺は揃っている）。図面タイトル・図番を `vs.SetObjectVariableString`（selector `1032`＝図面タイトル / `1033`＝図番）で設定し、`vs.UpdateVP` で描画を更新する。ビューポートが生成できない場合はシートレイヤだけ作って設定はスキップする。
- 実行順: シートはデザインレイヤを参照するため、`execute_document` はストーリ・通り芯・構造材・基礎の描画とレイヤ並べ替え（`reorder_story_layers`）を**すべて終えた後**に `execute_sheets` を呼ぶ。**並べ替えで床・野地板を最背面へ回した結果はインポート直後のビューポート描画には反映されない**（レイヤパレット・ビューポートのプロパティ上は新しい順なのに描画だけ古い順で床が柱梁を覆い隠す）。VW 上の検証で、並べ替えは既存ビューポートを out-of-date にせず、`UpdateVP`・`ReDrawAll`・`Project 2D` トグルによる強制再描画もスクリプト内では効かないことを確認した（手動「ビューポートを更新」またはファイルの開き直しでのみ反映される）。スクリプト内で自動反映する手段は VW の制約上無いため、余計な更新処理は行わずユーザーの手動操作に委ねる（VW の制約として許容）。
- **オブジェクト変数 selector（`SetVPLayerVisibility` の表示種別 0/1 を含む）は VectorWorks 公式のオブジェクト変数一覧・VW 上での実挙動に基づく**。シートレイヤ番号はレイヤ名（`CreateLayer` に渡す番号）が担う。シートレイヤタイトル=`159`・ビューポート図面タイトル=`1032` は VW で確認済み。ビューポート図番=`1033` は VW で最終確認する（描画フェーズは他の要素と同じく VectorWorks 上で検証する方針）。selector 値は `vw/sheet.py` 冒頭の名前付き定数（`_OV_SHEET_TITLE` 等）に集約しており、変更が必要な場合はそこを直す。

### 断面寸法データタグ（ifc/tag.py → vw/sheet.py）

床伏図・小屋伏図・母屋伏図のビューポート注釈として、各横架材の断面寸法（`120×180` 等）を表示する**データタグ**を配置する。ユーザーが VW 側で用意した `断面寸法` データタグスタイル（構造材の幅とせいを `120×180` 形式で表示する設定）を関連付け、対象の横架材オブジェクトにリンクする。タグは横架材の配置レイヤ（`layer`）を表示するビューポートに載るため、母屋・棟木（`n-母屋` レイヤ）のタグは母屋伏図に、それ以外の横架材（`n-横架材天端`／`n-軒高`）のタグは柱梁伏図に載る。

- 解析（`ifc/tag.py` の `build_tag_commands`）: **横架材の member 命令から**タグの配置を算出する（IFC を直接見ない）。横架材 1 本につき tag 命令 1 つ。`build_document` は横架材命令を一度だけ組み立てて `members` と `build_tag_commands(members)` の両方に渡す（tag 命令の `member_index` が `members` の並びと一致する必要があるため）。tag 命令は `style`（=`断面寸法`）・`layer`（横架材の配置レイヤ）・`member_index`（`members` 内インデックス）・`position`（挿入位置）・`angle`（文字角度）を持つ。
  - **向き**: 文字角度は横架材の軸方向（`start`→`end`）に合わせ、上下反転しないよう `(-90, 90]` に正規化する（`_tag_angle`）。
  - **配置側（上または左）**: 軸に直交する 2 候補（±90 度回転）から「上または左」を向く側を選ぶ（`_offset_side`）。y が大きい（上）方を優先し、y が同等な南北向きの材は x が小さい（左）方を選ぶ。これにより東西材は上辺、南北材は左辺にタグが並ぶ。
  - **位置**: 横架材の軸中央から、選んだ直交方向へ `断面幅/2` だけオフセットした**部材の辺の中央**（左右に伸びる梁は上辺中央、上下に伸びる梁は左辺中央）。ここに**データタグの下端中央**が来る。座標は他要素と同じグリッド中心オフセット済み（member 命令の座標をそのまま使う）。
  - **引き出し線を出さない**: タグを部材の面から離すと VW が関連付け先の横架材へ引き出し線を描くため、**余白を設けず面ちょうど（`断面幅/2`）に置く**。加えてデータタグの「引出線を表示」オブジェクトパラメータ（既定 ON）を描画フェーズで OFF にする（下記描画節。面に接して置くだけでは引出線パラメータが残るため）。
- 描画（`vw/sheet.py` の `execute_sheets` / `draw_tag`）: **タグはシート描画と同時に行う**。横架材レイヤは階ごとに固有なので、tag 命令の `layer` がビューポートの表示レイヤ（`viewport.layers`）に含まれる 1 枚の床伏図・小屋伏図にだけタグを載せる（基礎伏図や他階の伏図には載らない）。各タグは `vs.CreateCustomObject('Data Tag', position, angle)` でデータタグを作り、`vs.SetPluginStyle(obj, '断面寸法')` でスタイルを関連付け、**「引出線を表示」パラメータ（`_LEADER_FIELD`、既定 ON）を `vs.SetRField` で OFF（`_LEADER_OFF`）にして** `vs.ResetObject` で反映し、`member_index` が指す横架材ハンドル（`execute_members` が記録）に `vs.DT_AssociateWithObj` で関連付けてから、`vs.AddVPAnnotationObject(vp, obj)` でビューポート注釈に追加し `vs.DT_UpdateTaggedTags` で更新する。
  - **横架材ハンドルの受け渡し**: タグの関連付け先はモデル空間の横架材オブジェクト。ハンドルは横架材描画時にしか分からないため、`execute_members(commands, handles)` が配置した構造材ハンドルを**命令インデックスをキーに** `handles` dict へ記録し、`execute_document` がそれを `execute_sheets` へ渡す（フェーズ間は命令の並び順で対応付ける。JSON は順序を保つため `member_index` で一意に引ける）。フォールバック描画（プラグイン不可）でハンドルが無い横架材はタグを作るが関連付けは省く。
  - **データタグの内部プラグイン名**（`vw/sheet.py` の `_DATA_TAG_PLUGIN`＝`'Data Tag'`）・**引出線フィールド名/値**（`_LEADER_FIELD`＝`'Use Leader'`／`_LEADER_OFF`＝`'False'`）は VW が描画したデータタグの VectorScript エクスポートで確認済み。関連付けの挙動（注釈内タグがモデルレイヤのオブジェクトに関連付くこと）は **VW 上で最終確認する**（描画フェーズは他の要素と同じく VectorWorks 上で検証する方針）。

### 基礎伏図のグラフィック凡例（ifc/sheet.py → vw/sheet.py）

基礎伏図ビューポートに表示されるシンボル（既定ではアンカーボルト）の凡例として、VW 標準の**「グラフィック凡例」PIO** を基礎伏図のシートレイヤ上に配置する。凡例に載せるシンボルと表示ラベルは legend 命令の `items`（`{symbol, label}` の並び）が持ち、ラベルは**コード内の固定マッピング**で決める（要件どおり M12＝`土台用アンカーボルトM12`・M16＝`ホールダウン用アンカーボルトM16`）。

- 解析（`ifc/sheet.py` の `build_legend_commands`）: **アンカーボルトの anchor_bolt 命令から**凡例に載せるシンボルを決める（IFC を直接見ない）。`build_document` はアンカーボルト命令を一度だけ組み立てて `anchor_bolts` と `build_legend_commands(ifc_file, anchor_bolts)` の両方に渡す。基礎が無い（`footing.has_foundation` が偽＝基礎伏図が作られない）場合や、載せるシンボルが 1 つも無い場合は空リストを返す。**実際に配置されたアンカーボルトのシンボル**（`anchor_bolts` に現れるもの）のうち `ANCHOR_BOLT_LEGEND_LABELS` に対応があるものだけを `ANCHOR_BOLT_LEGEND_ORDER`（M12 → M16）の順で採る（入力順・重複に依存しない決定的な並び）。legend 命令は配置先シートレイヤ番号 `number`（＝基礎伏図 `1`）・シートレイヤ上の配置点 `position`（`FOUNDATION_LEGEND_POSITION`、VW 上で最終調整）・`items` を持つ。ラベルのマッピング（`ANCHOR_BOLT_LEGEND_LABELS`）・並び順（`ANCHOR_BOLT_LEGEND_ORDER`）・配置点は `ifc/sheet.py` 冒頭の名前付き定数に集約する。
- 描画（`vw/sheet.py` の `execute_sheets` / `draw_legend`）: **凡例はシート描画と同時に行う**。凡例はビューポート注釈ではなく**シートレイヤ上**に置くため、`draw_sheet` が返すシートレイヤハンドルを使い、legend 命令の `number` が一致するシートのシートレイヤ上に配置する。配置先シートレイヤ（番号＝レイヤ名）を `vs.Layer` でアクティブにし、`vs.CreateCustomObjectN('GraphicLegend', position, 0, False)` でグラフィック凡例 PIO を作る（第 4 引数 `showPref=False` でインポート中に設定ダイアログが開くのを防ぐ。柱束伏図記号 PIO と同じ）。**内部プラグイン名は表示名「グラフィック凡例」と異なり `GraphicLegend`（スペース無し）で、VW 上で実オブジェクトから確認済み（`GetParametricRecord` + `GetName`）。** 配置後、**ユーザーが VW 側で用意したグラフィック凡例スタイル `基礎伏図凡例`（`_GRAPHIC_LEGEND_STYLE`）を `vs.SetPluginStyle` で関連付ける**（構造材の `木質構造材_横架材`・データタグの `断面寸法` と同じプラグインスタイル方式）。**凡例のデータソース（＝シンボルをソースにし基礎伏図ビューポートでフィルタする設定）・集計基準・行レイアウトはこのスタイルが持つ**。グラフィック凡例は**矩形モード PIO** で、対話作成時はユーザーが描いた矩形の幅にレイアウトが追従する（サイズは OIP に出ない）。点で生成すると箱幅 0＝サイズ 0 でリサイズハンドルを掴めないため、**スタイル関連付けの後に**箱幅フィールド `BoxWidth`（`_LEGEND_WIDTH_FIELD`）を既定値 `_LEGEND_BOX_WIDTH`（シート上・ドキュメント単位 mm の適当な幅、VW で最終調整）に `vs.SetRField` で設定し `vs.ResetObject` で反映する（箱幅は by-instance のジオメトリでスタイルは決めない）。`BoxWidth` フィールド名は VW 上で実オブジェクトのパラメトリックレコードから確認済み。**凡例の実際の描画幅・高さはスタイルのソースから集めたセル（＝シンボル）の内容で決まる**ため、`SetPluginStyle` + `ResetObject` だけではセルが再計算されず凡例が空（セル 0 個＝幅 0 の縦線に潰れる）ままになる。そこで構造材・柱と同じく、**全配置後に `vs.UpdateStyledObjects('基礎伏図凡例')` を 1 回**呼んでスタイルのソースからセルをインスタンスへプッシュする（VW 上でスタイル編集ダイアログを開いて OK すると反映されるのと同じ再計算をこの呼び出しが担う。by-instance の `BoxWidth` 等は保持したまま by-style の内容のみ更新される。#56 と同種）。PIO が作れない（プラグイン未登録で NIL）・シートレイヤが作れない命令はスキップする。実行順はタグと同じくシート描画のループ内（`execute_sheets`）で、`UpdateStyledObjects` はループの後に 1 回だけ呼ぶ。
- **凡例のソース定義はプラグインスタイルに焼き込む**: グラフィック凡例のソース定義（`DefineSource`）は type=14 のボタンフィールドで、選択したソース（凡例ソース＝ビューポート・検索条件）を保持する文字列フィールドがパラメトリックレコードに存在しないことを**実オブジェクトの全フィールドダンプ（36 フィールド）で確認済み**（`SummarizeBy='StyleSymName'` 等の集計・並び替えはフィールドにあるが、ソース自体は `SetRField` で設定できない）。そのためソース定義（シンボルをソースにし基礎伏図ビューポートでフィルタ）を含めた凡例の設定はユーザーが VW 側でグラフィック凡例スタイル `基礎伏図凡例` として用意し、描画フェーズは PIO にこのスタイルを関連付けるだけにする。命令の `items`（シンボル → 固定ラベル）は凡例に載せる意図を記録として保持する。内部プラグイン名（`vw/sheet.py` の `_GRAPHIC_LEGEND_PLUGIN`＝`'GraphicLegend'`）・スタイル名（`_GRAPHIC_LEGEND_STYLE`＝`'基礎伏図凡例'`）は VW 側の登録名と一致させる必要がある。

## VectorWorks のレイヤとクラスの規則

- **通り芯のレイヤ**: すべて `共通` レイヤに配置。存在しない場合はスクリプトが自動作成。
- **通り芯のクラス**（X 通り・Y 通りの判定）:
  - 名前が `X` で始まる（大文字小文字不問）→ クラス `01作図-01線-01基準線-01通り芯-X通り`
  - 名前が `Y` で始まる（大文字小文字不問）→ クラス `01作図-01線-01基準線-01通り芯-Y通り`
  - それ以外: `|Δx| < |Δy|` の線（垂直に近い）を X 通り、それ以外を Y 通りとして扱う
- **GridAxis レコードフィールド**: `Label`（IFC から取得した軸名）、`ShowBubbleAt` = `"Start Point"`
- **構造材（柱・横架材）のクラス**（`ifc/structural_class.py`）: `04構造-02木造-…` 階層の葉クラス（番号 + 半角スペース + 名称を `-` で連結。例 `04構造-02木造-01土台-01土台`）を割り当てる。命令の `class` に格納し、描画フェーズ（`vw/member.py`・`vw/column.py`）が構造材オブジェクトに `vs.SetClass` で設定する（存在しないクラスは VW が自動生成）。判定方針:
  - **IFC 記録を信用する**: ホームズ君 IFC の `Name` には種別が埋め込まれる（`木梁:{種別}:{連番}` / `小屋束:…`、`STANDCOLUMN`）。横架材は `Name` 中央の種別トークンを `_MEMBER_CLASS_BY_TYPE` で直接クラスに対応付ける（`土台`→土台、`大引`→大引、`根太`→根太、`軒桁`→軒桁、`胴差`→胴差、`床小梁`/`床大梁`/`甲乙梁`→床梁、`小屋梁`→小屋梁、`母屋`→母屋、`棟木`→棟木）。柱は `ObjectType=STANDCOLUMN` または `Name` が `小屋束` 始まりなら小屋束。
  - **判別できないときは状況で推定する**（火打・隅木谷木・無名等）。横架材: 最下階=土台、中間階=床梁、最上階の軒高付近=小屋梁、軒高より高い（`above_eaves`＝天端が配置レイヤ高さを超える）=母屋。柱: 最上階の柱=小屋束、一般階は上下端の高さで判断し**上階の床（次階 FL）を貫けば通し柱、1 階分で止まれば管柱**（`is_through_column`、貫通判定の許容値 `THROUGH_COLUMN_TOL`）。
  - クラスは構造種別のみを表し、`member_id` の種別表記（柱の `管柱`/`小屋束` 等）とは独立。`member_id` は従来どおり `resolve_column_type`（通し柱を区別しない）で組み立てる。
- **基礎のクラス**（`ifc/footing.py`）: 立上り（壁）=`04構造-01基礎-03立ち上がり`、底盤・地中梁（スラブ）=`04構造-01基礎-02基礎スラブ`。命令の `class` に格納し、描画フェーズ（`vw/footing.py`）が `vs.SetClass` で設定する。

## 開発プロセス: PR 作成と監視

コード修正を実施する際は以下のプロセスに従う:

1. **PR作成の判断基準**:
   - コード編集後、ユーザーに確認すべき疑義が特にない場合は**自動的に PR を作成する**。
   - 迷いや未確定事項がある場合（変更方針をユーザーに確認中など）は、PR 作成を保留し先にユーザーに確認する。

2. **PR 作成後の対応**:
   - PR を作成したら `subscribe_pr_activity` で CI 結果とレビューコメントを監視する。
   - CI 失敗は原因を診断して修正コミットを自動的に push する。
   - レビューコメントは内容を確認し、軽微な修正は自動で追加コミットする。大きな変更・設計判断が必要な指摘はユーザーに確認してから対応する。
   - CI が全て green でレビュー上の問題もなければ**自動的にマージする**。

3. **コミットメッセージ**:
   - Claude セッション URL を追加する形式: `https://claude.ai/code/session_<SESSION_ID>`
